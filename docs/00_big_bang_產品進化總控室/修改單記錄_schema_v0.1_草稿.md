# 修改單記錄 Schema（v0.2 草稿）

> **目的**：定義「一張修改單到底存什麼」。這份記錄 = 精靈收集的東西 = 存進檔案的格式 = 未來資料庫的表。
> 精靈 UI、Excel 版面、檔名都是它的**下游**，可隨時改；**這份是唯一要謹慎、要鎖的東西。**
> **狀態**：v0.2 — 原 §8 五個開放問題已拍板，決議紀錄見 §9。（檔名沿用初版，內容以本標題版本為準。）

---

## 0. 設計原則（定錨用）

- 中間放乾淨的資料模型，邊邊放「笨笨的」轉換器（排版面 / 產各案編號）。
- 焊口存「**修改事件**」的語意，不存案場字串；字串只是輸出。
- 鐵底必填 = `before + after + 圖號PDF`；其餘每案用開關決定。
- 每筆帶**版本號**、原始值與算出值都存 → 可演進、可換 SQLite，不用重做。

---

## 1. 一張修改單（ChangeOrder）

| 欄位 | 說明 | 來源 |
|---|---|---|
| `schema_version` | `"0.2"` | 系統 |
| `id` | 單號＝`{流水號}_{date}_{當日序}`，如 `202_20260623_01` | 系統自動 |
| `status` | `草稿 / 待補 / 完整` | 系統 |
| `date` | 報告日期 `YYYYMMDD` | 使用者 |
| `series`（流水號） | 圖號的唯一短碼 | 使用者選 |
| `dwg_no`（圖號） | 完整圖號 | 由 `series` 帶出 |
| `scenario` | `normal / group`（大範圍整改、業主不出圖） | 自動判斷為主 |
| `welds[]` | 焊口事件清單 → §2 | |
| `photos[]` | 照片清單 → §3 | |
| `drawing_pdf` | 圖號 PDF + 標註 → §4 | |
| `reason` | 修改原因（文字） | 使用者 |
| `materials[]` | 結構化材料清單 → §5 | 使用者 |
| `authorization` | 業主簽認（**選填** → §6） | 使用者 |
| `audit` | **稽核軌跡 `history[]`**（每筆 who / when / action）——不是單一時間戳 | 系統 |

> 焊口清單存在記錄裡，**不塞進資料夾名**；資料夾名 = `id`，短而穩。
> `audit` 一開始就做成可擴充的 `history[]`（who/when/action）——「無 audit_trail」是本專案 P0 地雷，不可只存單一時間戳；需與既有 `operation_journal.py` 相容。

## 2. 焊口事件（weld event）

每一口都是一個**事件**，不是一個號碼：

| 欄位 | 說明 |
|---|---|
| `joint_type` | `焊口 / 管牙`（預設 焊口） |
| `origin` | `existing`（既有重焊）/ `new`（全新） |
| `base` | 原始焊口號（existing 用；new 為空） |
| `op` | `裁切 / 加長 / 縮短` |
| `rework_index` | 第幾次重工（→ 字母 a/b/c…）**程式自動算** |
| `code` | 算出的案場碼（如 `2a`、`1001`）**程式產生** |
| `spec.size / sch / material / weld_type` | 規格 |
| `spec_source` | `looked_up`（從管制表帶）/ `manual`（手填，多為 new） |

> **重複檢查不再是步驟**：existing 自動接下一個字母、new 自動取 `max+1`，天生不撞號。管制表在這裡只當「查規格 + 驗衝突」的唯讀來源。

---

## 3. 照片（photo）

| 欄位 | 說明 |
|---|---|
| `role` | `before`（問題長怎樣）/ `after`（修好的證明） |
| `file` | 檔名 |
| `weld_ref` | 對應哪一口（**選填**；照片預設綁整張單） |
| `annotations` | 圖上標註（選填） |

> 數量不限。鐵底：至少 1 張 `before` + 1 張 `after`。版面（精簡 / 多頁）由程式看數量自動決定，使用者不選。

---

## 4. 圖號 PDF

| 欄位 | 說明 |
|---|---|
| `file` | 對應圖號的 PDF |
| `annotations` | 在圖上「**圈位置**」——標哪裡改。內容（原因/改什麼/材料）由結構化欄位提供，**不重複輸入**；可選擇把結構化摘要蓋印到圖上。 |

---

## 5. 材料（material）— 結構化，不再攤平成純文字

`component / component_id / size / schedule / material / qty / unit / remark`
（沿用現精靈 Step6 已收集的欄位；差別只在：以前丟成 `materials.txt` 一行字，現在原樣存進記錄。）

---

## 6. 業主簽認（authorization）— **選填**

| 欄位 | 說明 |
|---|---|
| `approved_by` | 核准人 |
| `approved_at` | 核准時間 |
| `evidence` | 簽名的表單 / 簽認過的圖之照片或掃描 |

> 預設不擋——只要 before/after/pdf 的業主完全不受影響。需要稽核的案場，用 §7 的開關把它升成必填。

---

## 7. 每案設定（ProjectConfig，**獨立於記錄**）

- **編號方案**：字母規則、新焊口基數（這案 = 1000+）。
- **管制表**：路徑、欄名對應（含「焊 / 銲」變體）。
- **必填開關**：哪些欄位（如業主簽認、材料）在這個案場升級成必填。

> 同一份記錄，換一份 config 就能套到不同案場。這就是「統一代號」的真正落點。

---

## 8. 明確排除（不進這份記錄）

- 計量 / 請款 / RT% 那些（係數、RT口數、安裝計量列）——那是**另一個領域**。焊口管制表硬塞它們正是被詬病的點。本記錄只管「這一張修改單」。

---

## 9. v0.2 決議紀錄（原開放問題拍板）

1. **單號** = `{流水號}_{date}_{當日序}`；焊口清單進記錄、不進檔名。
2. **照片綁整張單**；`weld_ref` 設選填，給少數要指定某口的情況。
3. **PDF 標註**：結構化欄位為真相，標註只畫位置、可蓋摘要，不雙重輸入。
4. **業主簽認**：現在就留欄位、預設選填，案場可用開關升必填。
5. **管牙**：納入，焊口事件加 `joint_type`（焊口 / 管牙）。
