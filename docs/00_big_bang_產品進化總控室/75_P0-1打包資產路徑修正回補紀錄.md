# P0-1 打包資產路徑修正回補紀錄

日期：2026-06-22

## 背景

Checkpoint G 指出打包後 onedir 的程式資產位於 `_internal/`，但 `config.py` 的模板路徑仍指向 EXE 所在資料夾的 `template/`。因此 health-check 會過，但真正產出修改單時可能找不到樣板。

同類風險還包含少數模組用 `__file__` 推專案根，打包後可能把 records/settings 指到 `_internal/`。

## 本次修正

- 新增 `control/resources.py`
  - `resolve_project_dir()`：專案資料根。frozen 時為 EXE 所在資料夾。
  - `resolve_resource_dir()`：程式資源根。frozen 時為 `sys._MEIPASS` / `_internal`。
  - `project_path()`
  - `resource_path()`
- `control/config.py`
  - `BASE_DIR` 仍為專案資料根。
  - 新增 `RESOURCE_DIR`。
  - `TEMPLATE_PATH_6` / `TEMPLATE_PATH_27` 改走 `resource_path("template", ...)`。
- `settings_manager.py`
  - `settings.json` 改走 `project_path("settings.json")`。
- `material_pricebook.py` / `billing_batch.py`
  - records 內資料檔改走 `project_path(...)`。
- `material_constants.py` / `gui_annotator.py`
  - `wizard_data.json` 改走 `resource_path("control", "wizard_data.json")`。
- `wizard.py`
  - `wizard_data.json` 與材料圖片改走打包資源根。
- `tests/test_config_paths.py`
  - 補 frozen `_MEIPASS`、專案資料根與程式資源根分離測試。
- `diagnostics.py`
  - 診斷包加入 resource/template/wizard_data/seed 路徑與存在狀態。
- `check_release_package.py`
  - `--run-health-check` 時額外跑 packaged exe `--diagnostics`，確認 exe 自己看到的 `resource_dir` 是 `_internal`，且必要資產存在。

## 驗證目標

打包後：

- 專案資料仍寫入 EXE 所在資料夾。
- 模板、wizard_data、seed 等程式資產從 `_internal/` 讀取。
- release gate 後續才能真正檢查「可產出」，而不是只檢查資料夾結構。

## 本次實測

- 舊 dist 未重建時，`check_release_package --run-health-check` 會因 diagnostics probe 缺少新版資源路徑資訊而失敗，證明 gate 不再假綠。
- 重新執行 `python tools\build_release.py` 後，package gate 成功。
- packaged exe diagnostics probe 回報：
  - `resource_dir = dist\IEC-site-change-manager\_internal`
  - `template_6_exists = true`
  - `template_27_exists = true`
  - `wizard_data_exists = true`
  - `material_pricebook_seed_exists = true`
- packaged exe `--diagnostics` 可產生支援診斷包，啟動判斷為 `initialize`。
