<a name="殘酷現實檢查"></a>
## 殘酷現實檢查：先戳破四個會在導入時爆炸的假設

你要我「不要客氣」，那我先把最痛的講完。你現在的想法不是錯，而是**低估了從「我自己能用的工具」到「公司每天有人用、有錢經手的產品」之間的距離**。這段距離不是再多寫幾個功能，而是要補上一整層「就算使用者亂搞、檔案壞掉、兩個人同時開、Excel 沒裝、斷電當機，資料也不能爛、錢也不能算錯」的基礎建設。下面四點是會直接爆炸的。

### 假設一：「材料明細太簡易，之後再追蹤就好」

**現實：要追蹤「單價、金額、歷史」目前是 0%，而且更糟的是兩條路徑行為不一致。** 我實際追了程式碼：解析與寫入材料的邏輯（`parsers.parse_materials_txt()` + `record_manager.upsert_materials_rows()`）其實已經寫好、也已從舊檔遷出來了，但**只有 CLI 路徑（`main.py` 的 `run_cli`）會呼叫它**；大家實際在用的 GUI 路徑（`gui.py._process_folders`）從頭到尾沒有把結構化材料寫進 `records.json` 的 `materials[]`。所以實務上——使用者都是用 GUI——`materials[]` 永遠是空的（這也對得上 DATA_FLOW_AUDIT 的 L2：「嘗試讀但從未被寫入，永遠為空」）。而且就算走 CLI，`parse_materials_txt` 抓到的也只有「零件／尺寸／SCH／材質／數量／單位／備註」——**完全沒有單價、沒有金額、沒有料號**。換句話說：①現場填的材料多半只是 `materials.txt` 純文字躺在資料夾 ②就算被結構化也沒有任何價格資訊 ③CLI 跟 GUI 對材料的處理還不一樣（這正是後面會講的 `main.py`↔`gui.py` 重複邏輯沒同步的活案例）。要做到「追蹤單價、金額、歷史」，你基本上是從零開始。好消息是：你還沒被一堆髒資料綁死，現在重新設計成本最低。

### 假設二：「打包成單一 exe」就能變成公司級產品

**現實：單一 exe 不會讓你現在的三個隱形依賴消失，反而會把它們放大。**

1. **你依賴 Excel COM（pywin32）。** `excel_handler.py` 是用 COM 自動化去開 Excel、填模板、匯出 PDF。這代表**每一台要跑這支 exe 的電腦，都必須安裝、並授權啟用 Microsoft Excel**。COM 是單執行緒、會留殭屍 `EXCEL.EXE` 程序（你自己的 L5 風險就寫了 atexit 可能沒觸發）、不能並行、在背景服務或沒有桌面的環境直接死。把它包進 exe 不會解決，只會讓「為什麼這台電腦跑出來是空白 PDF」變成沒人能除錯的黑盒。
2. **你依賴外部 Excel 檔案。** 焊口管制表、DWG LIST 都是 settings.json 指向的外部 .xlsx。exe 包不進去，路徑還是會跑掉。
3. **settings.json 寫的是絕對路徑**（含磁碟代號）。你選了「雲端共用、多人多機」，這代表 A 的 `C:\Users\A\...` 到了 B 的電腦就是死路。這是 M5 風險，在多機情境下是 100% 會發生，不是「可能」。

exe 只是把 Python 直譯器包起來，**它打包的是程式，不是你的部署問題。**

### 假設三：「雲端共用資料夾，多人輪流用」這件事本身

**現實：Google Drive / OneDrive 同步資料夾 + 純 JSON 檔 + 沒有檔案鎖 ＝ 資料覆蓋的定時炸彈，而且是「無聲」爆炸。**

你的資料庫是 `records.json` 一個大檔。雲端同步軟體不是資料庫——它沒有鎖、沒有交易、沒有合併。實際會發生的事：A 在公司開著工具、B 在工地用另一台開同一個雲端資料夾，兩人都載入了 records.json 到記憶體，各自存檔。後存的整碗蓋掉先存的，**而且沒有任何錯誤訊息**。更糟的是 Google Drive 自己偵測到衝突時會產生 `records (1).json`、`records-A的電腦.json` 這種副本，你的程式根本不讀，資料就這樣裂成兩份。你目前 `records.json` 和 `billing.json` 都已經有原子寫入（`.tmp` + `os.replace`），這擋得住「單機寫一半斷電」——做得對。**但原子寫入完全不等於多人安全：兩個人各自「原子地」寫，後存的照樣「原子地」把先存的整碗蓋掉，而且沒有任何錯誤訊息。** 你缺的不是原子，是鎖。這是你現在最高優先、最危險、最容易被忽略的問題（你的 H4）。一旦 `billing.json` 開始記錢，這種無聲覆蓋就是直接的財務事故。（註：你的 `DATA_FLOW_AUDIT.md` 把 billing 標為非原子的 M4，那份是 3 月的掃描，程式碼後來已修成原子寫入——但「無鎖」這個更大的問題還在。）

### 假設四：「資料夾健康檢查」＝檢查那幾個資料夾在不在

**現實：資料夾在不在是最簡單的 5%。真正會讓程式產生錯誤報告、或當場崩潰的，是「半成品狀態（partial state）」。** 例如：records.json 說某張單有 before/after 照片，但實體檔案被人手動刪了；attachments 資料夾還在但 records 沒了（誤刪 records）；雲端同步到一半，note.txt 已同步但照片還沒下載完，工具讀到一個「看起來存在但是 0 byte」的圖；上次批次跑到一半當機，留下半張 Excel 和一個 `~$template.xlsm` 鎖檔。你問卷裡列的那些情境（誤刪、跑錯資料夾、損壞、升級）方向是對的，但**最危險的不是「缺東西」，是「東西在、但是是壞的或半成品的」**——這種狀態會通過「資料夾存在」的檢查，然後在後面某個讀檔的地方爆炸。

<a name="一產品定位與使用情境"></a>
## 一、產品定位與使用情境

### 1.1 這個工具到底該被定義成什麼？

先講你**現在實際上**做出了什麼，再講你**應該**把它定位成什麼。

現況：它是一個「**管線/焊口工程的修改單『產生器』**」——核心價值是把現場拍的 before/after 照片 + 焊口資料，自動填進 Excel 模板、產出制式 PDF 修改單。這件事它做得很好。

但你問卷裡想要的（材料追蹤、單價金額、請款狀態機、歷史查詢、多人共用、稽核）已經遠遠超出「產生器」。如果把這些都加上去，它的定位必須升級成：

> **「以修改單為最小單位的『工程變更與估驗請款資料管理系統』（Field Change & Billing Record System）」**

關鍵字是**資料管理系統**，不是填單工具。差別在於：填單工具產出檔案就結束了；**資料管理系統的產出物（PDF）只是副產品，真正的資產是那份結構化、可查詢、可稽核、可結算的資料。** 你一旦要追蹤錢，你就是在做資料管理系統，沒有中間地帶。

**我的建議定位（給你一句話對內溝通用）：**
> 「它是工地修改單的**單一真實來源（single source of truth）**：現場拍照填單 → 自動產出 PDF/Excel → 內業補齊材料 → 主管審核 → 請款結算，全程留痕可查。」

**定位上的取捨（請務必想清楚，因為它決定後面所有架構）：**

| 它是 | 它不該硬要變成 |
|---|---|
| 修改單的紀錄與請款管理 | 完整的會計/ERP 系統（別想取代公司的請款主系統，做好「提供乾淨資料給它」就好） |
| 工程現場的證據庫（照片、焊口、材料） | 全公司專案管理平台（別想管進度、甘特圖、人力） |
| 估驗/請款的「明細來源」 | 正式對外的請款單開立系統（除非財務同意，否則它產的是「請款依據」，不是法定單據） |

**【產品總監視角】** 最容易失敗的內部工具，都是因為「想做的太多」。你列的 12 個面向如果全部一次到位，會做兩年還上不了線。這份前導書的 Roadmap（第十一章）會逼你排序：**第 0、1 階段完全不碰新功能，只把現有的東西變得「不會爆」**，這比加任何功能都重要。

### 1.2 主要使用者與他們真正的工作流程

你提到的角色，我把它對應到實際情境，並標出每種人**最常做的 3 件事**。注意：你選的是「現場＋內業混合」，所以**現場端要極度防呆，內業端可以給較多欄位**——這是兩種完全不同的 UI 哲學，硬塞在同一個畫面是現在的隱憂。

| 角色 | 他是誰 | 最常用的 3 個工作流程 | 對這個人最重要的事 |
|---|---|---|---|
| **現場工程師/領班** | 在工地、用筆電或平板、手髒、趕時間、不太想打字 | ①拍照丟進收件匣 ②開新修改單（選日期/焊口/貼照片/選一句修改原因） ③確認單子產出來了沒 | **快、少打字、不要讓我選錯、錯了能改** |
| **內業/工程助理** | 在辦公室、整理資料、補材料、跑批次、對圖號 | ①批次產出 PDF ②補材料明細與單價 ③對焊口管制表/DWG、修正錯誤資料夾 | **看得到全貌、能搜尋篩選、能批次、能修正現場的錯** |
| **工地主管/工程師** | 要審核、要對數量、要簽 | ①檢視某段時間所有修改單 ②審核請款批次（這些單該不該請、金額對不對） ③看統計（這個月做了多少、請了多少、還沒請多少） | **可信、可追溯、看得出誰改了什麼** |
| **請款人員/內業會計介面** | 把修改單彙整成請款批次送出 | ①勾選可請款的單、建立請款批次 ②產出請款摘要與附件 ③追蹤退回/補件/已付款 | **不能重複請、不能漏請、退回要看得到原因、金額要有依據** |
| **公司會計（下游）** | 收到請款資料、入帳 | ①拿到一份乾淨、加總正確、附件齊全的請款明細 ②核對單價金額 ③確認沒有重複 | **資料乾淨、加總正確、可對帳、看得懂** |
| **系統管理者（很可能就是你）** | 安裝、設定路徑、升級、救火 | ①新電腦安裝/設定管制表路徑 ②版本升級與舊資料 migration ③出事時找 log、還原備份 | **安裝簡單、升級不毀資料、出事查得到原因** |

**【現場工務主管視角】** 我最怕的兩件事，你現在都還沒防：第一，**現場填的東西內業看不懂或要重問**（焊口碼、材質、原因不一致），這會變成每天的電話。第二，**單子產出來了，但請款時對不上**——因為現場根本沒填材料/數量，內業事後也補不齊。你要嘛在現場就逼出最低限度的可請款資訊，要嘛設計一條「內業補齊」的明確動線，不能兩邊都空著。

### 1.3 哪些流程要快、要安全、可以慢但要可追蹤

這是公司級產品的核心取捨，請貼在牆上：

**必須快（慢了現場就不用、改用紙本，工具就死了）：**
- 拍照 → 丟進收件匣（staging）→ 開新單貼照片。這條是命脈。現在的 7 步精靈 + staging 縮圖選取方向對，但 7 步對現場偏多（見第二章）。
- 「這張單產出來了嗎／長怎樣」的即時回饋。
- 焊口重複檢查要即時（你已經做到 O(1)，很好，保住它）。

**必須安全（錯了就是錢、就是對外文件、就是法律責任）：**
- 任何寫入 `records.json` / `billing.json` 的動作（多人覆蓋防護）。
- 請款狀態轉換（不能重複請、不能跳狀態）。
- 刪除/歸檔資料夾（要可復原）。
- 金額、單價、數量的編輯（要留痕、要二次確認）。

**可以慢，但一定要可追蹤（事後一定有人會問「這誰改的、什麼時候、為什麼」）：**
- 材料單價的歷史變動。
- 修改單從建立到請款到付款的完整生命週期。
- 資料 migration、批次作業、自動修復的紀錄。
- 設定的變更（誰把管制表路徑改掉了）。

**一句話總結這章：** 你現在的工具把「快」做得不錯，「安全」和「可追蹤」幾乎是空白。而公司級產品的價值，80% 在後面這兩塊。

<a name="二uiux-全面設計建議"></a>
## 二、UI / UX 全面設計建議

先講貫穿全章的三條原則，後面每個畫面都是它們的延伸：

1. **現場與內業是兩種人，不該共用同一個密度的畫面。** 現在四個 Tab（報告產出／紀錄管理／請款管理／設定）混在一起，欄位密度是給內業看的。現場應該有一個「大按鈕、少選項」的精簡入口。
2. **危險操作要嘛擋、要嘛能復原。** 現在到處是 inline 雙擊編輯（焊口、材料、請款狀態、金額），這對內業很爽，但**沒有 undo、沒有變更留痕**——現場手一滑改錯金額，沒人知道。
3. **空狀態、載入中、失敗狀態是三個真正的畫面，不是 console 的一行字。** 你現在這三個狀態大多是「沒有畫面」或「彈一個 QMessageBox」。公司級產品的成熟度，往往就看這三個畫面做得好不好。

### 2.1 逐畫面檢討

下面 18 個畫面，我標出**目的、該長怎樣、現在的問題、怎麼改**。

#### (1) 首次開啟畫面（Onboarding / 空專案）
- **該長怎樣**：一張卡片置中，標題「歡迎，這個資料夾還不是一個專案」，三顆大按鈕：「① 在這裡建立新專案」「② 我選錯資料夾了，重新選」「③ 從舊版升級／匯入」。下方一行小字顯示目前路徑（讓使用者確認自己在哪）。
- **現在的問題**：`啟動GUI.bat` 直接進主視窗，假設環境是好的。第一次在空資料夾跑，使用者看到的是一個空列表，不知道發生什麼事。
- **怎麼改**：啟動守門（第六章）先判斷狀態，再決定進哪個畫面。首次開啟**絕不能**直接進四 Tab 主畫面。

#### (2) 專案資料夾健康檢查畫面
- **該長怎樣**：一個檢查清單畫面，逐項打勾或打叉：`attachments/ ✅` `records/ ✅` `settings.json ❌ 缺失`，每個叉旁邊有「這代表什麼」和「建議動作」。底部兩顆按鈕：「自動修復可修復項目」「我自己處理」。**危險項目（如 records.json 損壞）要用紅色，且預設不自動修復。**
- **現在的問題**：`validator.py` 有驗證邏輯，但它是給「資料夾命名/照片齊不齊」用的，不是給「專案結構健康」用的，而且結果多半進 log 或 QMessageBox，沒有一個給人看的健康檢查畫面。
- **怎麼改**：把健康檢查做成一個正式畫面（第六章給完整規格），它是啟動流程的一部分。

#### (3) 主畫面資訊架構
- **該長怎樣**：頂部一條**專案標識列**（專案名 + 目前資料夾路徑 + 同步/鎖定狀態燈），讓人永遠知道「我在哪個專案、現在安不安全」。主體維持 Tab，但 Tab 命名與順序要照工作流：**收件匣 → 修改單 → 請款 → 報表 → 設定**。
- **現在的問題**：①頂部沒有「我在哪個專案」的常駐資訊，多專案/多機時很容易在錯的資料夾改東西。②「報告產出」當第一個 Tab，是工具導向不是任務導向的命名。③設定藏了一堆只有你懂的東西（焊口管制表動態欄位）。
- **怎麼改**：加常駐專案列（含鎖定狀態，見第七章）。Tab 改任務導向命名。設定分「一般使用者設定」與「進階/管理者設定」兩層。

#### (4) 修改單列表（紀錄管理）
- **該長怎樣**：一個可搜尋、可篩選、可排序的表格 + 右側詳情側欄。每列最重要的欄位：日期、流水號、焊口摘要、**狀態徽章（草稿/已產出/待請款/請款中/已付款/退回）**、縮圖。狀態用顏色（見 2.4）。
- **現在的問題**：`RecordManagerPanel` 有搜尋，但**狀態欄位根本不存在於資料模型**——目前一張單沒有「生命週期狀態」，只有「有沒有產出 PDF」。所以列表無法一眼看出「哪些待處理」。「未產出標記」是靠掃 attachments 比對，是 derived 的、慢且脆弱。
- **怎麼改**：資料模型加 `status`（第三章），列表第一眼就是狀態驅動。把「未產出」變成一個正式狀態而不是即時掃描推斷。

#### (5) 修改單新增／編輯流程
- **該長怎樣**：**新增用 wizard，編輯用詳情側欄（不要用 wizard）。** 新增是線性、一次性、引導式；編輯是局部、隨機存取、改一個欄位就好。
- **現在的問題**：①新增是 7 步 wizard（日期序號 → 模式 → 焊口 → 照片 → 原因 → 材料 → 確認），對現場偏長，且第 1 步「日期+流水號」要使用者懂流水號規則。②編輯走 `EditRecordDialog`，但它做的事很重（改焊口會連動重命名資料夾、改 weld_info.json、改 GroupWeld.txt），這種「改一個欄位觸發一連串檔案系統操作」的動作**沒有預覽、沒有確認清單、沒有 undo**，風險很高。
- **怎麼改**：
  - 把 7 步壓成感知上的 3 大步：**「這是哪張單」（日期/序號/模式自動帶）→「拍了什麼」（焊口+照片）→「為什麼改」（原因+材料）**。材料在現場可「先跳過、內業補」。
  - `EditRecordDialog` 存檔前出一個**「即將執行的變更清單」**：「資料夾將從 A 重命名為 B、weld_info.json 將更新、GroupWeld.txt 將更新」，要使用者確認；並且這整組操作要能 rollback（第七章）。

#### (6) 附件照片管理畫面
- **該長怎樣**：before/after 並排對照，大圖、可 hover 放大（你已有）、可替換/新增/刪除，且每張圖顯示「檔案是否真的存在、解析度、是否疑似 0 byte/同步未完成」。
- **現在的問題**：①`_HoverThumb` / `_ZoomLabel`（gui_panels）與 wizard 的 `_ZoomPopup` 是兩套重複實作（你自己標的風險），行為不一致。②沒有偵測「雲端同步未完成的假檔」。③替換照片是即時生效、無 undo。
- **怎麼改**：統一一套縮圖/zoom 元件放 `theme.py`。讀圖時檢查檔案大小與可解碼性，對「0 byte/解不開」的圖顯示「☁️ 雲端同步中或檔案損壞」而不是裂圖。替換/刪除照片要進「可復原」機制（至少先移到 `.trash/` 而非真刪）。

#### (7) 材料明細登記畫面（重點，詳見第四章）
- **該長怎樣**：一個表格，但**每列輸入要像填表單一樣被引導**：品名用可模糊搜尋的下拉（帶出單位、預設單價）、規格用相依下拉、數量用數字鍵盤友善輸入、金額自動算。底部即時小計。
- **現在的問題**：`Step6_Materials` 與紀錄管理的「材料明細 tab」是 inline 自由輸入，**沒有料號、沒有單位標準化、沒有單價**，而且如前述**根本沒寫進結構化資料**。這是現在最大的功能空洞。
- **怎麼改**：見第四章三方案。重點是「現場好填、內業可查、財務可算」。

#### (8) 請款追蹤畫面（詳見第五章）
- **該長怎樣**：上方狀態統計卡（你有了），中間是**請款批次**列表（不是單張修改單列表），點進批次看裡面勾選的單。狀態用色票。每個批次能產請款摘要 PDF/Excel。
- **現在的問題**：`BillingPanel` 目前是「每張單一個金額 + 一個狀態下拉 + inline 編輯」，**沒有『批次』這個概念**，也沒有狀態機。這跟真實請款流程（一次送一批、會被退、會部分付）對不起來。
- **怎麼改**：見第五章，導入「請款批次」為一級物件。

#### (9) PDF / Excel 輸出畫面
- **該長怎樣**：一個「產出工作台」：左邊勾選要產的單、右邊即時進度（哪張在跑、哪張跳過、哪張失敗），失敗的能一鍵重試、能看原因。產完給「開啟資料夾／開啟 PDF」。
- **現在的問題**：你已經有背景執行緒 + 進度條 + 指紋跳過 + 失敗重試（`_process_folders` / `_retry_failed`），這塊其實做得不錯。**主要缺的是失敗的『原因』對使用者可讀**——現在失敗大概只說「失敗」，使用者不知道是 Excel 沒裝、模板缺、還是照片壞。
- **怎麼改**：失敗項目要分類顯示原因（見 2.3 與第八章 Excel 依賴）。

#### (10) 錯誤提示與修復畫面
- **該長怎樣**：錯誤訊息三段式：**發生什麼（人話）→ 為什麼（可選展開技術細節）→ 我現在能做什麼（具體按鈕）**。例如不是「PDF 匯出失敗 (COMError -2147…)」，而是「這台電腦的 Excel 沒有回應，無法產出 PDF。你可以：①重試 ②略過這張稍後再產 ③回報給管理員（已自動附上 log）」。
- **現在的問題**：錯誤大多是技術訊息直接彈 QMessageBox，對現場/內業是天書。
- **怎麼改**：建一個 `UserError` 包裝層，把技術例外翻譯成「現象+建議動作」。這是現場人員能不能自救、還是每次都打給你的關鍵。

#### (11) 設定頁面
- **該長怎樣**：分兩層。**一般設定**（路徑就用「選資料夾」按鈕、圖片品質用「省空間/標準/高畫質」三檔，不要露 1280/quality 85 這種數字）。**進階/管理者設定**（焊口管制表動態欄位、完整性檢查等級、debug）用一個「我是管理員」展開或密碼保護。
- **現在的問題**：`SettingsPanel` 把「焊口管制表動態欄位 CRUD」這種只有你懂的東西跟基本路徑混在一起，現場人員一進去就懵。`HelpTooltip` 系統是加分，但解決不了資訊架構問題。
- **怎麼改**：一般 vs 進階分層。一般設定盡量「零數字、零術語」。

#### (12) 稽核紀錄 / 操作紀錄畫面（目前不存在，必須新增）
- **該長怎樣**：一個唯讀的時間軸表格：時間、誰（電腦名/使用者）、做了什麼（建立單 72、把 71 狀態從待請款改成已付款、刪了一張照片）、可篩選可搜尋。
- **現在的問題**：**完全沒有。** 你有 `logs/app.log`（rotating file），但那是給工程師除錯的技術 log，不是給主管看「誰動了什麼」的業務稽核。多人共用 + 經手錢，沒有稽核軌跡是公司級產品的致命缺口。
- **怎麼改**：第三章加 `audit_trail` 模型，這個畫面是它的視圖。

#### (13) 批次作業畫面
- **該長怎樣**：所有「會一次動很多筆」的操作集中在這（批次產 PDF、批次補登焊口、批次預處理圖片、批次改狀態）。每個批次作業前顯示「即將影響 N 筆」，作業中顯示進度，作業後顯示「成功 X、跳過 Y、失敗 Z」摘要 + 可匯出結果。
- **現在的問題**：批次能力散在各處（產出在 Tab1、補登在 dialog、預處理在按鈕），心智模型不統一，且批次「結果摘要」不一致。
- **怎麼改**：統一「批次作業」的互動模式：預覽影響 → 執行 → 結果摘要 → 可復原（能復原的話）。

#### (14) 空狀態畫面（Empty State）
- **該長怎樣**：每個列表「沒資料」時，不是空白，而是「這裡還沒有 X，要不要 ①建立 ②匯入 ③了解這是什麼」。例如請款 Tab 沒資料時：「目前沒有任何修改單進入請款流程。修改單要先標記為『可請款』才會出現在這裡。」
- **現在的問題**：空列表就是空列表，使用者不知道是「還沒有」還是「壞了沒載到」。
- **怎麼改**：每個主要列表都設計專屬空狀態文案 + 引導動作。

#### (15) 載入中畫面（Loading）
- **該長怎樣**：區分「快（<0.5 秒不顯示）」「中（顯示 spinner + 在做什麼）」「慢/批次（顯示進度條 + 可取消）」。讀大量 records、掃 attachments、跑 Excel COM 都可能慢。
- **現在的問題**：載入 records、掃描 attachments 找「未產出」可能在 UI 執行緒做，造成卡頓假死，使用者以為當機。
- **怎麼改**：耗時操作一律背景執行緒 + 明確「正在掃描 172 個資料夾…」文案。**永遠不要讓主視窗無回應超過 1 秒而沒有任何提示。**

#### (16) 失敗狀態畫面（Failure）
- **該長怎樣**：跟錯誤畫面(10)呼應，但這是「整個操作失敗」的版面，例如「批次產出 23 張，其中 3 張失敗」，把 3 張列出來、各自原因、各自能做什麼。
- **現在的問題**：失敗多半是一個彈窗講完就沒了，使用者關掉就忘了哪幾張要重弄。
- **怎麼改**：失敗清單要「留在畫面上、可重試、可匯出」，不是彈窗即焚。

#### (17) 權限不足 / 鎖定狀態畫面
- **該長怎樣**：當專案被別台電腦鎖住（第七章 lock），顯示「這個專案目前由 **〔電腦名/使用者〕** 於 **〔時間〕** 開啟編輯中，你現在是**唯讀模式**。可以①唯讀檢視 ②要求對方關閉 ③強制接管（危險，需確認）」。當檔案是雲端唯讀/沒寫入權限，明確說「你對這個資料夾沒有寫入權限」。
- **現在的問題**：**完全沒有鎖、也沒有這個畫面**。多人共用情境下，這是你從「會偶爾掉資料」進化到「保證不掉資料」的關鍵畫面。
- **怎麼改**：第七章 lock 機制 + 這個唯讀/鎖定畫面，是雲端共用部署的**必做**項。

#### (18) 升級 / Migration 畫面
- **該長怎樣**：偵測到舊版資料時：「偵測到舊版專案（資料版本 1.x），需要升級到 2.x 才能使用新功能。升級前會自動備份。」→ 進度 → 結果。**升級必須先備份、可回滾、且不可逆操作前明確警告。**
- **現在的問題**：你有 `migrate_excel_to_json()`，但沒有面向使用者的「升級流程畫面」，也沒有「資料版本號」驅動的升級判斷（settings 有 version 但沒拿來 gate 升級）。
- **怎麼改**：見第六、八章的 migration 設計。

### 2.2 哪些按鈕會讓使用者誤會（具體點名）

| 易誤會的元素 | 為什麼誤會 | 建議 |
|---|---|---|
| 到處的 **inline 雙擊編輯** | 使用者不知道「雙擊就能改」，或相反——**手滑雙擊就改了還不知道** | 編輯改為明確的「編輯」鍵或鉛筆 icon；inline 編輯要有明顯的「編輯中」外框與「儲存/取消」 |
| **「儲存變更」** 按鈕（紀錄管理） | 同時動 records.json + weld_info.json + 可能重命名資料夾，但按鈕只寫「儲存」 | 改成「儲存（將更新 N 筆紀錄與資料夾）」或存檔前出變更清單 |
| **「歸檔」 vs 「刪除」** | 歸檔是移到 `_archived/`，使用者常以為是刪除、或以為刪除是真刪 | 文案講清楚「歸檔＝收起來，之後可還原」「刪除＝（如果有）移到垃圾桶」；危險色區分 |
| **「清空 staging」** | 一鍵刪除收件匣所有原始照片，**這是不可逆的真刪** | 必須二次確認 + 顯示「將刪除 N 個檔案」+ 最好先移到 `.trash/` |
| **「自動寫入焊口管制表」** | 會回寫到一個共用的外部 Excel，影響別人 | 要明確「這會修改大家共用的管制表」，並記錄是誰寫的 |
| **「恢復預設值」（設定）** | 可能把使用者設好的管制表路徑也清掉 | 區分「重置執行參數」與「重置全部」，後者要強確認 |
| **PDF/Excel 的「產出」** | 使用者分不清「產出 PDF」跟「匯出 Excel 紀錄清單」 | 動詞統一：產出修改單(PDF) vs 匯出資料(Excel)，icon 區分 |

### 2.3 主畫面該露什麼、不該一直露什麼

**該放主畫面（第一眼）：** 專案名與路徑、同步/鎖定狀態、待辦數量（幾張草稿、幾張待請款、幾張失敗待處理）、最近活動。

**不該一直露（收進進階/設定/右鍵）：** 焊口管制表動態欄位設定、圖片預處理數值、完整性檢查等級、debug 模式、指紋/快取這些內部概念。**使用者不該知道「指紋」「快照」「快取」這些字。**

### 2.4 互動規格速查（直接拿去當設計準則）

| 主題 | 準則 |
|---|---|
| **需要二次確認的操作** | 刪除/清空 staging、強制接管鎖定、升級資料、批次改狀態、改已送出的請款、回寫共用管制表 |
| **應允許復原（undo / 還原）** | 刪除照片、刪除/歸檔修改單、改金額與狀態、批次操作、重命名資料夾 |
| **應該用 wizard（線性引導）** | 新增修改單、首次設定、資料升級 |
| **應該用表格（隨機存取/比較）** | 修改單列表、材料明細、請款批次、稽核紀錄 |
| **應該用詳情側欄（不換頁編輯）** | 點一張修改單看/改細節、看一個請款批次內容 |
| **需要搜尋** | 修改單（焊口碼/流水號/日期/原因）、材料（品名/料號）、稽核紀錄 |
| **需要篩選** | 狀態、日期區間、是否已請款、是否有照片缺失 |

| **需要標籤（tag）** | 退件原因、急件、待補材料、待補照片 |

### 2.5 怎麼讓不太碰電腦的現場人員看得懂

1. **用工地的話，不要用程式的話。** 「指紋未變更已跳過」→「這張沒有改，不用重做」。「fingerprint/staging/snapshot」這些字現場人員一個都不該看到。
2. **大按鈕、少選擇、給預設。** 現場 90% 的操作就是「開新單、貼照片、選一句原因」。把這條路做成首頁三步，其他都收起來。
3. **錯誤要給出口，不要給代碼。** 每個錯誤畫面都要有「我現在按哪顆」。
4. **顏色 + 圖示 + 中文，三重編碼。** 不要只靠顏色（有人色弱），狀態同時用顏色、icon、文字。
5. **不可逆的事一律攔一下。** 現場手忙腳亂，刪錯、改錯是常態，預設假設「使用者會手滑」。
6. **照片是現場的主語言。** 列表、確認頁、請款摘要都盡量帶縮圖，現場人員靠看圖確認比看文字快。

**【UX 視角一句話】** 你現在的 UI 是「給懂這個系統的人用的」——它假設使用者知道流水號規則、知道雙擊能編輯、知道歸檔不是刪除、看得懂技術錯誤。公司導入後第一週，這些假設每一條都會變成一通電話打給你。

<a name="三資料模型設計"></a>
## 三、資料模型設計

### 3.0 先做一個架構決定：JSON 檔還能撐多久？

這必須先講，因為它決定下面所有東西怎麼存。你現在是「一個 `records.json` 大檔 + 幾個附屬 JSON」。我的建議分三段，**對應你「現在雲端共用、未來可能上 DB」的選擇**：

- **第 0～1 階段（現在）：留在 JSON，但把它做到「防彈」。** 原因見下方「SQLite 陷阱」。短期內把原子寫入、檔案鎖、schema 驗證、一致性檢查補滿，比換儲存引擎更急。
- **第 2 階段：每個專案改用一個 `project.sqlite`（單檔 SQLite）。** 當材料、請款、稽核的關聯變多，JSON 的「整碗讀進來、整碗寫回去」會在效能與一致性上撐不住。SQLite 提供交易（transaction）、外鍵約束、查詢，且仍是「單一檔案」好攜帶。
- **第 3 階段（你說的未來進化）：中央伺服器 + PostgreSQL/MySQL + API。** 當要跨專案統計、權限控管、即時多人，才值得這個成本。

> **⚠️ SQLite 陷阱（吹毛求疵但很重要）：** 很多人一聽「該上資料庫」就想把 SQLite 檔放進 Google Drive / OneDrive 共用資料夾。**這是已知的資料庫損壞陷阱。** 雲端同步軟體會在 SQLite 寫到一半時複製檔案、會把 `-wal`/`-shm` 旁檔跟主檔分開同步、會產生衝突副本，導致整個 DB 損壞且難以修復——比 JSON 還慘（JSON 至少壞了還看得懂、救得回）。**所以：只要你還在「雲端資料夾同步」這個部署模式，就不要把 SQLite 放進同步資料夾。** 真正要上 SQLite 的安全前提，是搭配「單寫者鎖」+「同步時整個專案打包，不要讓同步軟體碰半開的 DB」，或乾脆等到第 3 階段直接走中央 DB。這正是「該上資料庫」這個直覺會爆炸的地方。

**下面的資料模型用『邏輯模型』描述（欄位、關聯、約束），它同時適用 JSON 與 SQLite。** 我會標出每個實體的唯一鍵與防重策略，這些不管哪種儲存都要遵守。

### 3.1 命名與通用欄位約定

- 每個實體都有 `id`（系統產生、永不重用、永不給使用者看的內部主鍵，建議 UUID 或 `日期-序號-亂數`）。**不要用「日期+資料夾名」當主鍵**——它會因為重命名而變動，現在你的 records 正是用這個當邏輯主鍵，所以一改焊口要連動改一堆地方。
- 每個實體都有稽核四欄：`created_at`、`created_by`、`updated_at`、`updated_by`（`by` = 電腦名/使用者名，多人共用必備）。
- 狀態類欄位一律用**固定字串列舉**（enum），不要用中文自由字串（現在請款狀態是中文字串，容易因為一個空格或同義字裂開）。
- 金額一律存**整數（分）或 Decimal 字串**，**絕不用 float**（float 會有 0.1+0.2 問題，財務不可接受）。

### 3.4 這章最關鍵的三個提醒

1. **主鍵用不變的 id，不要用會變的「日期+資料夾名」。** 這是你現在所有「改一個欄位連動改一堆檔案」痛苦的根源。
2. **金額用整數分或 Decimal，狀態用 enum，價格存當下快照。** 這三條是財務資料不出錯的底線。
3. **加 audit_trail 與 migration_version 這兩個現在完全沒有的實體。** 沒有前者，多人經手錢的系統不可稽核；沒有後者，升級舊資料是賭博。

<a name="四材料明細設計"></a>
## 四、材料明細設計

這是你點名的重點，我發散講。先重述殘酷現實：**現在材料在實務上沒有可用於計價的結構化資料**（GUI 路徑不寫 `materials[]`、CLI 寫的也只有零件/數量、無單價金額），所以這比較接近「從白紙設計，而且要一次設計對」。設計目標你說得很好：**現場好填、內業可查、財務可算**。這三者其實互相拉扯，下面就是怎麼平衡。

### 4.1 核心矛盾與破解

| 矛盾 | 現場要的 | 財務要的 | 破解法 |
|---|---|---|---|
| 欄位多寡 | 越少越好，最好只填數量 | 要料號/單位/單價/金額 | **分階段填**：現場填「品名+數量」，內業/系統補單價金額 |
| 自由 vs 標準 | 想打什麼打什麼 | 要能 group by 統計 | **常用料下拉帶出標準欄位，允許手動但標記 is_manual，事後可轉正** |
| 單價 | 不想碰 | 要當下正確價 | **料庫帶入預設價，存成單據快照，不回溯** |

### 4.2 欄位設計：最小可用 vs 專業版

| 欄位 | 極簡(現場必填) | 平衡 | 專業 | 怎麼來 |
|---|:--:|:--:|:--:|---|
| 品名 name | ✅ | ✅ | ✅ | 可模糊搜尋下拉（料庫）+ 允許手打 |
| 數量 qty | ✅ | ✅ | ✅ | 數字輸入，數字鍵盤友善 |
| 單位 unit | 自動帶 | ✅ | ✅ | 料庫帶入（支/m/個/kg…），可改 |
| 規格 spec | — | ✅ | ✅ | 相依下拉（選了品名才出規格） |
| 材質 type | — | ✅ | ✅ | 下拉 SS/CS…（你 wizard_data 已有） |
| 料號 code | — | 自動 | ✅ | 料庫帶，手動料先空 |
| 單價 unit_price | — | 內業補 | ✅ | 料庫帶當下價，可覆寫（留痕） |
| 金額 amount | — | 自動 | ✅ | qty×unit_price 自動算 |
| 類別 category | — | 預設「材料」 | ✅ | 材料/工資/耗材/雜項 |
| 備註 note | — | 選填 | ✅ | 自由 |

**「哪些可以先隱藏」：** 料號、單價、金額、類別在現場模式預設隱藏；切到內業模式才出現。**「哪些由常用材料帶入」：** 單位、規格選項、材質選項、預設單價。**「哪些用下拉」：** 品名（可搜尋）、單位、材質、規格、類別。**「哪些可模糊搜尋」：** 品名與料號（打「彎」出 Elbow、打「90」出 90° 彎頭）。

### 4.3 棘手細節的處理建議

- **單位：** 一定要標準化清單（支/m/mm/個/PCS/kg/組/式），不要讓人自由打，否則統計時「支」「隻」「pcs」「PCS」會裂成四種。料庫每個品項綁預設單位。
- **數量：** 允許小數（管長 2.5m），但整數件數限整數；負數要擋（除非做退料）。
- **單價：** 三層來源——①料庫當下價（預設）②本專案合約價（覆寫料庫）③手動。**存進單據時一律快照當下值**，料庫日後改價不回溯既有單據。
- **歷史價格：** `price_history` append-only，查得到「2026-03 這料是多少」。報表可選「用當下價」或「用最新價」重估。
- **稅金：** 不要混進每列材料。**稅是『單據/批次層級』的事**，在請款批次層算（未稅小計 → 稅 → 含稅）。每列材料存未稅單價。
- **工資/耗材/雜項：** 用 `category` 欄位區分，跟材料同一張表但可分類加總。工資列可以沒有料號。
- **材料群組：** 用 catalog 的 `category` + 可選 `group`（如「管件」「閥類」「焊材」），報表能按群組小計。你的 `wizard_data.json` 已經有 components（Pipe/Elbow/Tee/Reducer…）含 image/icon，這是很好的群組基礎，直接長成料庫骨架。
- **同材料不同規格的混亂：** 用「品名 + 規格」當料庫的業務唯一鍵。Elbow 90° 2" SS 和 Elbow 45° 2" SS 是兩個 catalog 項，不是一個項自由填規格——否則統計永遠對不齊。
- **手動材料轉正：** 現場手打的料標 `is_manual=true`，內業在料庫管理頁看到「待轉正清單」，可一鍵「建為正式料」並回填料號到所有引用它的單。
- **避免每次都填很多：** ①最近用過的料置頂 ②「複製上一張單的材料」③常用組合做成「套餐」（一個「2吋不鏽鋼接管」套餐帶出管+兩彎頭+焊材）。
- **避免日後資料太髒：** 關鍵就是「現場能手打、但手打的都進待轉正佇列被內業收斂」，髒資料有出口被清理，而不是擴散。
- **匯入 Excel：** 料庫支援從 Excel 匯入（欄位映射，你 settings 已有動態欄位映射的成熟做法可複用）。單張單的材料也可貼 Excel。
- **從舊資料補齊：** 你已經有 `parsers.parse_materials_txt()` 能把 `materials.txt` 解析成結構化欄位（零件/尺寸/SCH/材質/數量/單位），不用重寫——把它接進 GUI 流程、再補上單價/料號對應即可。解析不出的標「待人工確認」。

### 4.4 三種方案

**方案 A — 極簡（現場 30 秒填完）**
- 只有：品名（下拉/手打）+ 數量 + 單位（自動）。無單價無金額。
- 優點：現場零阻力。缺點：財務算不了錢，內業要全部補單價。
- 適合：完全不想讓現場碰錢的公司。

**方案 B — 平衡（推薦）⭐**
- 現場填：品名 + 數量（+ 規格/材質可選）。內業補：單價（料庫帶預設，可改）→ 金額自動。料庫 + 手動料轉正機制。稅在請款批次層算。
- 優點：現場負擔低、財務拿得到可算的資料、髒資料有收斂出口。缺點：要做料庫與「現場/內業」雙模式。
- 適合：**你的情況（現場＋內業混合）。**

**方案 C — 專業（全欄位 + 核價流程）**
- 全欄位 + 多級單價（料庫/合約/手動）+ 核價狀態 + 群組套餐 + 完整歷史重估報表。
- 優點：接近專業估驗系統。缺點：開發成本高、現場學習曲線陡。
- 適合：第 2～3 階段成熟後再上。

**我的推薦：方案 B，但料庫從一開始就照方案 C 的 schema 設計（欄位先留著、UI 先隱藏）。** 這樣現場用起來是 B 的輕量，未來要升 C 不用改資料結構，只要把隱藏欄位打開。**最貴的錯誤是現在用 A 的簡陋 schema，半年後要補單價歷史時發現資料全是自由文字、無法回填。**

### 4.5 給現場的材料輸入畫面（具體長相）

> 一列一料。點「+ 加材料」→ 跳出一個搜尋框，打字即時濾出料庫（帶 icon，你已有 31 張零件圖）。選中後自動帶單位、規格選項。只要填數量。右側即時顯示「本單材料 8 項」。內業模式下，每列右邊多出單價欄與金額（自動算），底部多一條未稅小計。**現場看不到錢，內業看得到錢，同一份資料兩種視圖。**

<a name="五請款追蹤設計"></a>
## 五、請款追蹤設計

先講定位：你現在的 `BillingPanel` 是「一張單一個金額、一個中文狀態下拉、inline 編輯，存進空的 billing.json」。**這距離一個真實的請款流程，缺的是『批次』這個核心概念與一台狀態機。** 真實工程請款不是一張一張請，是「這個月把這 20 張湊成一批估驗送出去，業主退了 3 張，補件後重送，分兩期付款」。下面照這個現實設計。

### 5.1 兩層狀態機

**請款有兩個層級的狀態，不要混在一起：** 修改單自己的請款狀態（billing.status）、以及請款批次的狀態（batch.status）。

**(A) 單張修改單的請款狀態機**
```
                        ┌──────────────────────────────────────┐
                        ▼                                        │
[草稿] ──產出PDF──▶ [已產出] ──標記可請款──▶ [可請款] ──納入批次──▶ [請款中]
                                                  ▲                  │
                                                  │             ┌────┼─────────┐
                                          (退回/補件後)          ▼    ▼         ▼
                                                  └──[退回]  [部分請款]  [已請款]
                                                       │         │         │
                                                   (補件重送)     └────▶ [已付款]
                                                                          │
[任何狀態] ──作廢──▶ [作廢]（終態，需權限+原因）        [已付款]──────────┘(終態)
```

**狀態轉換規則（防呆核心）：**
- `草稿 → 已產出`：必須有 before/after 照片、必要焊口資料齊全。
- `已產出 → 可請款`：必須有材料金額（否則請款沒有依據）。**這一步逼出「沒填材料就不能請款」的約束**，解決你現場不填材料的根因。
- `可請款 → 請款中`：被納入一個「編輯中/已送出」的批次時自動轉。
- `請款中 → 退回`：批次裡這張被業主退，記 `reject_reason`。
- `退回 → 可請款`：補件後可重新納入批次。
- `請款中 → 部分請款 / 已請款`：依批次回款。
- `→ 已付款`：終態。
- `→ 作廢`：需要權限與原因，終態，且**作廢不是刪除**（資料留著、可稽核）。

**(B) 請款批次的狀態機**
```
[編輯中] ──送主管審──▶ [已送審] ──主管核可──▶ [已送出(給業主)] ──┬──▶ [已收款](終態)
   │                       │                                   ├──▶ [部分付款] ──▶[已收款]
   │                  (主管退回)                                └──▶ [退回] ──(修正)──▶[編輯中]
   └──作廢──▶[作廢]
```

### 5.2 完整流程逐步拆解（對應你列的每一點）

| 你的問題 | 設計答案 |
|---|---|
| **修改單何時可請款** | 通過「有照片 + 有焊口 + 有材料金額」三道門，狀態才能進「可請款」。沒到位的單根本進不了請款列表。 |
| **如何建立請款批次** | 在請款 Tab 按「新增批次」→ 選業主/期數 → 進入批次編輯畫面。batch_no 系統產（原子，防兩人撞號）。 |
| **如何批次勾選修改單** | 批次編輯畫面左側列出所有「可請款」的單（含縮圖、金額），勾選加入。加入即從「可請款」轉「請款中」。 |
| **如何避免重複請款** | **硬約束：一張單同時只能存在於一個『進行中』批次。** 已在某批次的單，在別的批次列表會變灰、標「已在批次 B-2026-03」。資料層 `billing_item` 的 `(record_id, 進行中)` 唯一。 |
| **如何處理部分請款** | `billing_item.ratio` 或 `amount` 表達。一張 10 萬的單先請 6 成 → 這張進「部分請款」，剩 4 成可在下一期再開 item。 |
| **如何處理退回修改** | 批次或單被退 → 狀態 `退回` + 必填 `reject_reason` → 單退出批次、回「可請款」前先停在「退回」讓人看到。 |
| **如何處理補件** | `退回 → 補件中 → 可請款`。補件動作記錄補了什麼（補照片/補材料/改金額），進稽核。 |
| **如何處理已送出未付款** | 批次「已送出」、單「請款中」。可記預計收款日，逾期可在報表標紅。 |
| **如何處理已付款** | 批次「已收款」→ 底下 items 轉「已付款」。可記實收金額（對帳用，可能與請款額有差）。 |
| **如何處理作廢** | 需權限 + 原因。作廢單/作廢批次都保留資料、進稽核，列表用刪除線灰顯示，不參與統計。 |
| **如何連結 PDF/Excel/附件** | 批次的「請款摘要」自動蒐集每張單的 PDF、材料明細，可一鍵打包成一個資料夾或合併 PDF 給業主。 |
| **如何產生請款摘要** | 批次層產出：封面（業主/期數/總額/含稅）+ 逐單明細（單號/焊口/材料/金額）+ 附件索引。Excel 與 PDF 兩版。 |
| **如何查某張單的請款歷史** | 單的詳情側欄有「請款歷程」時間軸：何時進哪個批次、被退過幾次、何時付款，全來自 audit_trail + billing_item 歷史。 |
| **如何讓主管審核** | 批次「已送審」狀態，主管視圖只看「待我審」的批次，能看明細、退回（附原因）或核可。 |
| **如何讓會計看得懂** | 匯出的請款明細：一列一單、未稅/稅/含稅分明、加總自動且鎖死、附單號可回查、無任何自由格式。 |

### 5.3 欄位、防呆、稽核、報表

**關鍵防呆：**
- 加總永遠由系統算，**人不能手改總額**（只能改單列金額）。
- 金額用整數分/Decimal。
- 「送出」批次前檢查：每張單都有金額、都有 PDF、沒有缺照片。
- 改「已送出/已付款」的批次要二次確認 + 權限 + 留痕。
- 刪批次不是真刪，是作廢。

**稽核（進 audit_trail）：** 建批次、加入/移除單、改金額、送審、核可/退回、送出、收款、作廢——每一步記「誰、何時、改了什麼」。

**報表需求：**
- 請款狀態總覽（未請/請款中/部分/已請/已付各多少張、多少錢）——你已有統計卡，擴充成這個。
- 某業主/某期的請款明細。
- 應收未收（已送出未付款）逾期表。
- 某段時間的請款 vs 付款對帳表。
- 材料用量統計（跨單，靠第四章結構化材料才做得到）。

### 5.4 給你一個現實提醒

**【財務請款顧問視角】** 你最大的風險不是「功能不夠」，是「金額沒有依據」。如果現場/內業不填結構化材料，請款金額就是內業憑印象打的一個數字，業主一問「這 8 萬怎麼算的」就答不出來、就退件。所以**第四章的材料結構化，是第五章請款能不能成立的前提**——這兩章是綁在一起的，不能只做請款不做材料。另外，**這個工具產的請款明細，定位是『請款依據/估驗計算稿』，不是公司正式對外的請款發票**。請先跟你們會計確認：它是要餵資料給現有請款系統，還是要直接當對外單據？這決定要不要做到發票級的嚴謹（編號控管、不可竄改、列印格式法遵）。

<a name="六啟動守門與資料夾健康檢查"></a>
## 六、啟動守門與資料夾健康檢查

這章是把「殘酷現實假設四」展開。核心理念：**exe 啟動後，第一件事不是開主視窗，是『搞清楚我現在站在什麼樣的資料夾上』，再決定給使用者看什麼。** 啟動流程應該是一個獨立的「守門人（gatekeeper）」階段。

### 6.1 啟動守門總流程

```
exe 啟動
  │
  ├─0. 取得目前工作資料夾（exe 所在/被開啟的資料夾）
  │
  ├─1. 找 .project.json？
  │     ├─ 沒有 → 這不是專案 → 看資料夾像不像「跑錯了」or「全新空資料夾」（見 6.3）
  │     └─ 有 → 讀 schema_version
  │
  ├─2. schema_version vs 程式版本
  │     ├─ 相同 → 進健康檢查(3)
  │     ├─ 較舊 → 升級流程（先備份→migration→驗證）
  │     └─ 較新 → 「這個專案是用更新版本建立的，請更新工具」擋住（防舊版毀新資料）
  │
  ├─3. 結構健康檢查（6.2）：必要資料夾/檔案、partial state、雲端同步未完成
  │     ├─ 全綠 → 取得專案鎖(第七章) → 進主視窗
  │     ├─ 可自動修復項 → 提示 →（使用者同意）修復 → 重檢
  │     └─ 危險項（records 損壞等）→ 停在健康檢查畫面，不自動修，給選項
  │
  └─4. 取得單寫者鎖：拿到 → 可編輯；拿不到 → 唯讀模式（2.1-17 畫面）
```

### 6.2 偵測規則、文案、修復策略對照表

| 情況 | 偵測規則 | 給使用者的話（文案） | 修復策略 |
|---|---|---|---|
| **第一次開新專案** | 沒有 `.project.json`，且資料夾空或只有零星檔 | 「這個資料夾還不是專案，要在這裡建立新專案嗎？」 | 使用者確認後，建立全部必要資料夾 + settings + .project.json（**自動，安全**） |
| **跑錯資料夾** | 沒有 `.project.json`，但資料夾裡有一堆不相干檔案（如桌面、下載夾特徵） | 「這看起來不像工務修改單的專案資料夾。你目前在：〔路徑〕。要改選其他資料夾嗎？」 | **不自動建任何東西**，引導重選（避免在桌面亂建一堆資料夾） |
| **必要資料夾缺失** | `attachments/ records/ output/ pdf/ staging/ logs/` 任一缺 | 「缺少 N 個必要資料夾。」逐項列 | 缺空資料夾＝**可自動修復**（建空的沒風險） |
| **settings.json 缺失** | 檔案不存在 | 「設定檔不見了，將用預設值重建（路徑需要重設）。」 | 自動用預設重建，但**標記路徑需使用者重設**（不要亂猜路徑） |
| **records.json 損壞** | 檔案存在但 JSON 解析失敗 / schema 驗證失敗 | 「主紀錄檔損壞，這是嚴重狀況。已找到 N 個備份，最近一個是〔時間〕。」 | **絕不自動覆蓋。** 列出 `backups/` 與 `.bak.*`，讓使用者選還原哪個；提供「以唯讀方式嘗試讀取損壞檔搶救」 |
| **attachments 在、records 不在** | 有照片資料夾，records.json 不存在或為空 | 「偵測到 N 個資料夾但沒有紀錄。可能是紀錄檔遺失，或這是從別處複製來的資料。」 | 提供「**從 attachments 重建 records**」（掃描重建，**需使用者確認**，因為會產生大量推斷資料） |
| **records 在、attachments 遺失** | records 指向的資料夾不存在 | 「N 筆紀錄找不到對應的照片資料夾。可能被刪除或雲端尚未同步完成。」 | **先別動**。可能是雲端還沒同步（見下）。提供「標記為遺失」而非刪除紀錄 |
| **output/pdf/staging/logs 缺** | 資料夾不存在 | （通常不用驚動使用者）「重建產出資料夾。」 | **可自動修復**（這些是衍生物） |
| **多個 exe 同開** | 本機 lock 偵測到同程式另一實例 | 「工具已經開著了。」 | 切到既有視窗，不開第二個 |
| **多人共用同資料夾** | 專案鎖被別台持有 | 「此專案正由〔電腦〕編輯中，你是唯讀。」 | 唯讀模式 / 要求接管（第七章） |
| **雲端同步未完成** | 偵測到同步軟體佔用檔、或檔案 size=0/部分、或有 `.tmp`/衝突副本 | 「偵測到雲端可能正在同步，部分檔案可能還沒下載完。建議稍候再操作。」 | **不自動修復**，建議等待 + 提供「重新檢查」按鈕 |
| **寫入中斷/崩潰殘留** | 存在 `*.tmp`、`~$*.xlsm`、lock 過期、半完成標記 | 「上次似乎沒有正常關閉。」 | 清理 Excel 暫存鎖（`~$`）＝可自動；`.tmp` 比對後可清；殘留 lock 過期可接管 |

### 6.3 「不自動修復」的危險清單（很重要）

**這些情況絕對不要自動修復，自動修會把小事變大事：**
1. `records.json` / `billing.json` 損壞 → 自動覆蓋＝直接毀掉唯一資料。永遠讓使用者選備份還原。
2. records 指向的 attachments 不見 → 自動刪紀錄＝可能只是雲端沒同步完，刪了就真沒了。
3. 偵測到「跑錯資料夾」→ 自動建專案＝在使用者桌面/下載夾亂拉屎。
4. schema 較新 → 自動降級＝新版資料被舊版毀掉。
5. 任何「無法判斷是誤刪還是正常空」的狀況 → 一律問人，不要替使用者決定。

**原則一句話：可自動修復的，都是「建立衍生物/空容器」這種加法且無損的動作；只要牽涉覆蓋、刪除、或推斷使用者意圖，一律停下來問。**

### 6.4 備份策略

- **寫入前備份**（你已有 `.bak.*` 與 `backups/` 保留 5 份）——但**批次處理時 5 份不夠**（L4），改成「按次數 + 按時間」雙保留（最近 5 次 + 每天第一次，保留 7 天）。
- **升級前強制完整備份**整個 `records/`（含 billing）到 `backups/pre-migration-{version}-{時間}/`，可一鍵回滾。
- **備份要含 billing.json**（現在的自動備份只顧 records，錢的資料反而沒被備份，是漏洞）。
- 雲端情境額外建議：每天自動把 `records/` 壓一份到 `backups/daily/`，因為雲端的「版本歷史」對 JSON 內部覆蓋不一定救得回。

### 6.5 lock / transaction / staging 策略（與第七章呼應）

- **專案鎖（lockfile）**：取得編輯權時在 `records/.lock` 寫入「電腦名 + 使用者 + PID + 心跳時間」。心跳每 30 秒更新；超過 N 分鐘沒心跳視為過期可接管。這是雲端多人的**必做**。
- **交易式寫入**：所有「一個動作改多個檔案」（如改焊口連動重命名資料夾 + weld_info + GroupWeld）要包成可回滾的交易（第七章）。
- **staging 是緩衝不是儲存**：收件匣的東西在「分派進某張單」前都是暫態，清空要進 trash。

**【導入顧問視角】** 啟動守門做得好不好，直接決定你會不會「每天接電話救火」。90% 的現場求救會是：「我打開都是空的」（跑錯資料夾/雲端沒同步）、「它說檔案壞了」（同步衝突）、「我的資料不見了」（被另一台覆蓋）。把這章的健康檢查 + 鎖 + 不亂自動修做紮實，等於把你自己從 24 小時技術支援裡解放出來。

<a name="七防重複寫入與資料一致性"></a>
## 七、防重複寫入與資料一致性

這章是你雲端共用部署的「保命章」。把所有會造成資料重複、覆蓋、半成品、不一致的情況，逐一給對策。

### 7.1 防重複/覆蓋對照表

| 風險 | 會怎麼發生 | 對策 |
|---|---|---|
| **修改單編號重複** | 兩人同時建單拿到同一個 report_no / 流水號 | report_no 產生要在**取得鎖的情況下原子配號**；建立前再查一次唯一性；用不變 id 當真正主鍵，編號重複至少不會撞毀資料 |
| **附件資料夾重複** | 同一 (日期,流水號) 被建兩次；或群組/single 命名撞名 | 建資料夾前檢查存在性；存在則提示「已存在，要開啟既有的還是建立 _2」 |
| **PDF 重複產生** | 來源沒變卻一直重產；或兩人同時產同一張 | 用 `source_fingerprint`（你已有指紋）跳過未變更；產出寫入用原子（先寫 .tmp 再 rename）；產出加鎖 |
| **請款重複** | 同一張單被放進兩個批次 | 第五章硬約束：一張單同時只能在一個「進行中」批次（資料層唯一約束 + UI 變灰） |
| **材料重複** | 同一料在一張單列很多次、手動料與料庫料重複 | 同 (catalog_id, spec) 提示合併；手動料轉正時去重 |
| **多人同時寫入** | 雲端兩台各自存檔，後者蓋前者 | **單寫者鎖 + 心跳**（6.5）。沒拿到鎖只能唯讀。這是根治 H4 的關鍵 |
| **同一人連點兩次** | 雙擊「建立/產出/送出」按鈕觸發兩次 | 操作中禁用按鈕（防連點）；關鍵操作做冪等（用 id 判斷已執行就略過） |
| **程式當機後重開** | 留下半張 Excel、半個 .tmp、過期 lock | 啟動清理殘留 `.tmp`/`~$`；過期 lock 可接管；半完成的產出靠 fingerprint 重判 |
| **雲端同步延遲** | 讀到還沒下載完的檔（0 byte/部分） | 讀檔前驗 size 與可解碼；偵測同步中提示稍候（6.2） |

### 7.2 交易式寫入與 rollback（產生一半怎麼辦）

**問題情境（你現在真實存在）：** `EditRecordDialog` 改焊口時，會「重命名資料夾 + 改 weld_info.json + 改 GroupWeld.txt + 改 records.json」。如果做到第 3 步斷電，資料夾已改名但 records 還是舊名 → 對不上 → 下次讀檔崩潰。

**對策：把「一個邏輯操作」包成交易：**
1. **先寫，後切換（write-then-swap）**：所有檔案先寫到暫存，全部成功後才一次性 rename 上線；任一步失敗就刪暫存、什麼都沒變。
2. **操作日誌（journal）**：開始多檔操作前，寫一個 `.journal` 記「我要做 A→B→C」；做完刪掉。啟動時若發現殘留 journal，代表上次沒做完，依 journal 回滾或續做。
3. **records.json 與 billing.json 都已有原子寫入**（`.tmp`+`os.replace`）——很好；確保**所有** JSON 寫入（dwg_map、weld_snapshot、wizard_data 等附屬檔）都走同一機制，別留漏網的直接覆蓋。原子只防斷電，多人覆蓋要靠上面的鎖。
4. **「先進垃圾桶不真刪」**：刪除/重命名前，把原始物件複製到 `.trash/{時間}/`，確認成功才清。

### 7.3 一致性檢查（把你的 _audit_data.py 變成系統的一部分）

你已經有一支很好的 `_audit_data.py`（一次性掃描），和 `utils.check_integrity()`。**把它升級成系統內建、可隨時跑、啟動可選跑的 `audit_integrity()`**（你 DATA_FLOW_AUDIT 的建議 A1）：
- records ↔ attachments 雙向對應（有紀錄沒資料夾 / 有資料夾沒紀錄）。
- details/material/photo → record 外鍵完整（沒有孤兒）。
- report_no、(date,serial) 唯一性。
- billing_item 沒有「一張單在兩個進行中批次」。
- 金額加總一致（batch.total = Σitem.amount）。
- 結果分「資訊/警告/錯誤」三級，錯誤給「如何修」。

### 7.4 操作歷史的保留

- **audit_trail**（第三章⑯）是業務層歷史：誰改了什麼。append-only、永不刪。
- **logs/app.log**（你已有）是技術層歷史：程式怎麼跑的。兩者分開，不要混。
- 重要：**多人共用時 audit 的 actor 要能識別到「哪台電腦/哪個 Windows 使用者」**，否則「誰改的」永遠是謎。

**【架構師視角】** 這章如果只能做三件事：①單寫者鎖 + 心跳（根治多人覆蓋）②所有 JSON 原子寫入 + 多檔操作交易化（根治半成品）③內建 audit_integrity（讓問題被「發現」而不是「爆炸」）。這三件做完，你的資料安全等級會從「單人堪用」跳到「多人敢用」。

<a name="八產品化與-exe-打包"></a>
## 八、產品化與 exe 打包

這章回答「要變成公司級 exe 還缺什麼」。先說結論：**打包技術不是你的瓶頸，依賴與部署才是。**

### 8.1 最大的產品化障礙：Excel COM 依賴（必須先做決定）

`excel_handler.py` 用 pywin32 COM 自動化 Excel 來填模板、匯出 PDF。這是你產品化最大的單一風險：

| 問題 | 後果 |
|---|---|
| 每台電腦都要裝 + 授權 Microsoft Excel | 沒裝 Excel 的電腦直接不能用；授權是公司成本與合規問題 |
| COM 是單執行緒、會留殭屍程序 | 你自己的 L5：atexit 沒觸發會留 EXCEL.EXE；批次/並行更危險 |
| COM 在不同 Excel 版本行為不一 | 「在我電腦好的，在你電腦空白」這種無法遠端除錯的鬼故事 |
| 無法在無桌面/服務環境跑 | 未來要上中央伺服器自動產出時，COM 直接死路 |

**三條路，請選一條（建議混合）：**
- **短期（第 0-1 階段）：保留 COM，但加防護**——產出全程 try/finally 確保關閉 Excel、偵測「沒裝 Excel」給明確訊息（不是 COMError）、單例排隊不並行。
- **中期（第 2 階段）：改用 openpyxl 直接寫 .xlsx**（你已依賴 openpyxl），脫離 Excel 安裝。難點是你的模板是 .xlsm 且要插圖、要轉 PDF——openpyxl 能寫值與插圖但**不能執行巨集、不能自己轉 PDF**。轉 PDF 改用 LibreOffice headless（`soffice --convert-to pdf`）或專門的 PDF 產生，可在任何機器/伺服器跑。
- **長期（第 3 階段）：報表改用程式直接生 PDF**（reportlab/WeasyPrint），完全脫離 Excel，模板變程式控制，最穩、最可測、可上伺服器。

**這是你「單一 exe」夢想能否成真的關鍵**：只要還綁 Excel COM，你的 exe 就不是「下載就能跑」，而是「下載 + 裝 Excel + 設好路徑才能跑」。

### 8.2 打包策略

- **PyInstaller** 仍是 PyQt6 最務實選擇（`--onefile` 或 `--onedir`）。**建議 `--onedir` 而非 `--onefile`**：onefile 每次啟動要解壓到 temp，慢且更容易被防毒攔；onedir 啟動快、好除錯、好做差分更新。
- **PyQt6 + PyMuPDF + Pillow + pywin32** 都有原生 binary，PyInstaller hook 要測；PyMuPDF/PyQt 常見漏打包，要實機驗證。
- **你 requirements 同時有 pypdf 與 PyPDF2**（L3）——選一個（建議 pypdf），移除另一個，減少打包體積與版本衝突。

### 8.3 各種檔案放哪（打包後路徑會變，這是常見地雷）

| 東西 | 放哪 | 理由 |
|---|---|---|
| **程式/依賴** | exe 內（onedir 的 `_internal/`） | 唯讀，跟著版本 |
| **templates（.xlsm）、icon、零件圖** | 打包進 exe（用 `sys._MEIPASS` 或 onedir 路徑解析） | 屬於程式資產，不該讓使用者改 |
| **使用者設定 settings.json** | **專案資料夾內**（現狀，保留） | 每專案不同 |
| **全域/機器設定**（如這台電腦的 Excel 路徑） | `%APPDATA%/工務修改單/` | 跨專案、跟使用者走 |
| **logs** | 專案內 `logs/` + 另存一份到 `%APPDATA%`（崩潰時專案可能不可寫） | 雙保險 |
| **公司主料庫** | 未來中央化；現階段可放共用唯讀位置 | 跨專案共用 |

**關鍵：程式碼裡所有路徑都要用「相對 exe」或「相對專案根」解析，絕不能有硬編碼絕對路徑。** 你 settings 的絕對路徑（M5）+ weld_snapshot.json 裡我看到的 `"attachments_root": "C:\\Users\\a0976\\..."` 都是這類地雷，打包到別台就死。

### 8.4 版本、更新、migration

- **版本號**：採語意化版本（app 版本 `2.3.1`）+ 獨立的**資料 schema 版本**（`schema_version`）。兩者分開——程式可以小改而 schema 不動。
- **顯示版本**：關於頁要顯示 app 版本 + schema 版本 + build 時間，現場回報問題時你才知道他用哪版。
- **自動更新需不需要？** 內部工具初期**不建議做自動更新**（複雜、易出包）。建議：①放一個共用磁碟/雲端的「最新版」資料夾，啟動時比對版本提示「有新版」②給一鍵下載。等使用者數多了再考慮做靜默更新。
- **舊資料 migration**：由 `schema_version` 驅動，每個版本寫一支冪等 migration，升級前強制備份、可回滾（第六章）。**migration 一定要有測試**（第九章）。

### 8.5 Windows 權限 / 防毒 / 安裝

- **權限**：不要寫入 `Program Files`（需要管理員）。程式放使用者目錄或讓使用者選安裝位置；資料放專案夾/APPDATA。避免任何需要 UAC 提權的行為。
- **防毒誤判**：PyInstaller onefile 常被誤判為病毒（解壓行為 + 未簽章）。對策：①用 onedir ②**程式碼簽章憑證（code signing）**——這是公司級散佈幾乎必備，能大幅降低 SmartScreen/防毒攔截 ③首發前在公司常見防毒上實測。
- **公司內部安裝**：做一個 Inno Setup / MSI 安裝包，比「丟一個 exe 給大家」專業且好管理（建捷徑、設關聯、列在「新增移除程式」）。或用公司的軟體派送（SCCM/Intune）。

### 8.6 使用者不懂 Python 時怎麼排錯

- **內建「回報問題」按鈕**：一鍵把最近 log + 環境資訊（版本/路徑/Excel 有無）+ 目前專案健康檢查打包成一個 zip，讓使用者寄給你。**這一顆按鈕能省你 80% 的來回問診。**
- **錯誤訊息人話化**（2.1-10）：把例外翻成「現象 + 建議動作」。
- **自我診斷頁**：設定裡放一個「環境檢查」：Excel 有沒有、路徑通不通、寫入權限 OK 不、雲端同步狀態，綠燈/紅燈一目了然。
- **絕不讓 Python traceback 直接彈到使用者臉上**——那是你現在會嚇跑現場人員的東西。

**【導入顧問視角】** 「打包成 exe」聽起來是終點，其實是起點。真正讓工具能在公司活下來的是：簽章（不被防毒擋）、安裝包（IT 好派送）、回報按鈕（你不用一台台跑）、版本顯示（出事知道哪版）、脫離 Excel 依賴（不用每台裝 Office）。這些沒有一個是「功能」，但少了任何一個，導入都會卡住。

<a name="九測試策略"></a>
## 九、測試策略

你已有 100 個測試（parsers/utils/record_manager/staging），核心邏輯有覆蓋，這是很好的起點。但**GUI、Excel COM、多人併發、migration 這些「最會出事的地方」目前 0 覆蓋**。下面照風險排優先序。

| 層級 | 測什麼 | 現況 | 優先 |
|---|---|---|---|
| **單元測試** | parsers、utils、record_manager CRUD、staging | ✅ 已有 100 個 | 維持，每改必跑 |
| **資料模型測試** | 唯一鍵、外鍵完整性、狀態機合法轉換、金額用 Decimal 不失真、enum 驗證 | ❌ 無 | 🔴 高（財務正確性） |
| **一致性測試** | audit_integrity 能抓出孤兒/重複/對不上 attachments；故意造髒資料看抓不抓得到 | ❌ 無 | 🔴 高 |
| **請款流程測試** | 狀態機所有轉換、防重複請款（一單兩批次要被擋）、部分請款加總、退回補件、作廢不參與統計 | ❌ 無（功能還沒做） | 🔴 高 |
| **材料統計測試** | 單位標準化、金額加總、群組小計、歷史價快照不被回溯 | ❌ 無 | 🟡 中 |
| **併發/檔案鎖測試** | 模擬兩個程序搶鎖、心跳過期接管、唯讀模式不可寫 | ❌ 無 | 🔴 高（雲端核心） |
| **中斷寫入測試** | 在 write-then-swap 各階段 kill，驗證資料不壞、可回滾、journal 能續做 | ❌ 無 | 🔴 高 |
| **migration 測試** | 用真實舊版資料樣本，升級後資料不丟、冪等（跑兩次一樣）、可回滾 | ❌ 無 | 🔴 高 |
| **雲端資料夾測試** | 模擬同步延遲（0 byte/部分檔/衝突副本）、唯讀權限、`.tmp` 殘留 | ❌ 無 | 🟡 中 |
| **大量資料測試** | 1000+ records 的載入/搜尋/掃描效能（你現在 UI 執行緒掃 attachments 會卡） | ❌ 無 | 🟡 中 |
| **錯誤資料測試** | 損壞 JSON、缺欄位、編碼問題、超長字串、特殊字元焊口碼 | 部分 | 🟡 中 |
| **PDF/Excel 產出測試** | 模板填值正確、插圖位置、PDF 合併、跳過未變更、失敗重試；**最好脫離 COM 後才好自動測** | ❌ 無（COM 難測） | 🟡 中 |
| **UI 操作測試** | 關鍵流程（建單 wizard、改狀態、建批次）用 pytest-qt 跑；二次確認/防連點有效 | ❌ 無 | 🟢 低-中 |
| **打包後 exe 測試** | 在乾淨 Windows（沒裝 Python、模擬沒裝 Excel）跑 exe；路徑解析、資產載入、防毒 | ❌ 無 | 🔴 高（上線前必做） |

**測試策略三原則：**
1. **錢與資料安全的測試（資料模型、請款、併發、中斷、migration）優先於 UI 測試。** 介面醜還能用，金額算錯或資料覆蓋是事故。
2. **exe 要在「乾淨機器」測**——你自己的電腦什麼都裝了，測不出別人的問題。準備一台/一個 VM 沒裝 Python、沒裝 Excel 的環境。
3. **COM 難自動測，這正是該脫離 COM 的另一個理由**——脫離後產出邏輯才能進 CI 自動跑。

<a name="十一開發-roadmap"></a>
## 十一、開發 Roadmap

四階段。**第 0、1 階段刻意不加大功能，只把現有的變成「不會爆」**——這是最反直覺但最重要的紀律。

### 第 0 階段：現有工具安全收束（目標：不要壞、不要重複寫、不要掉資料）

- **目標**：在不加新功能的前提下，把資料安全做到「多人雲端共用也不會掉」。
- **要做**：①單寫者鎖 + 心跳 + 唯讀模式畫面（records/billing 主檔已原子，但無鎖＝多人仍會互蓋，這是最急的）②附屬 JSON（dwg_map/weld_snapshot 等）也補原子寫入 ③多檔操作交易化 + journal + .trash ④settings/snapshot 絕對路徑改相對 ⑤備份策略升級（**含 billing**、批次夠用）⑥內建 audit_integrity（啟動可選跑）⑦清棄用碼。
- **不做**：材料系統、請款批次、新 UI、打包優化。
- **驗收**：模擬兩程序搶寫不覆蓋；任意階段 kill 不壞資料、可回滾；換一台機器/改磁碟代號仍可用；audit_integrity 能抓出人造髒資料。
- **風險**：鎖實作不當會「鎖死沒人能編輯」——心跳過期接管要設計好。
- **開發順序**：原子寫入/交易 → 鎖 → 路徑相對化 → 一致性檢查 → 備份 → 清理。

### 第 1 階段：公司內部可穩定使用（目標：一般人每天用，你不用在旁救火）

- **目標**：現場 + 內業能每天自助使用，錯誤能自救。
- **要做**：①啟動守門 + 健康檢查畫面（第六章全套）②錯誤訊息人話化 + 「回報問題」一鍵打包 ③現場精簡模式（3 大步建單）④空/載入/失敗/唯讀狀態畫面 ⑤records↔attachments 一致性即時維護 + 雲端假檔偵測 ⑥生命週期狀態（草稿/已產出/可請款…）⑦Excel COM 防護（沒裝 Excel 給明確訊息、確保關閉）。
- **不做**：請款批次完整流程、材料料庫（先把狀態與守門做穩）、脫離 COM。
- **驗收**：找 2-3 個真實現場/內業使用者試用一週，求救次數可量化下降；沒裝 Excel 的機器給的是人話不是 COMError；故意跑錯資料夾/斷網同步，工具不崩且給對的提示。
- **風險**：現場模式若還是太複雜，使用者會退回紙本——要實地測。
- **開發順序**：守門/健康檢查 → 錯誤人話化/回報 → 狀態欄位 → 現場精簡模式 → 各狀態畫面。

### 第 2 階段：資料管理成熟（目標：材料、請款、附件、PDF、紀錄可追蹤可統計）

- **目標**：從「產生器」正式變成「資料管理系統」。
- **要做**：①材料系統方案 B（料庫 + 現場/內業雙模式 + 手動轉正 + 歷史價快照）②請款批次完整流程 + 兩層狀態機 + 防重複請款 + 請款摘要 ③材料/請款報表與統計 ④audit_trail + 稽核畫面 ⑤（評估）每專案改 SQLite ⑥（評估）脫離 Excel COM 改 openpyxl + LibreOffice/程式產 PDF。
- **不做**：中央伺服器、跨專案、權限角色（除非請款需要基本審核權限）。
- **驗收**：能回答「這個月用了多少某材料、總額多少」「這張單請款歷程」「應收未收有哪些」；請款防重複請被測試證實；舊 materials.txt 能被補齊。
- **風險**：材料/請款是最大工程，容易做不完——嚴格用方案 B 的範圍，別滑向方案 C。
- **開發順序**：材料料庫 → 材料登記 UI → 請款狀態機 → 請款批次 → 報表 → 稽核 →（DB/脫 COM 視情況）。

### 第 3 階段：專業產品化（目標：打包、升級、回報、權限、備份、維護完整）

- **目標**：能像正式軟體一樣散佈與維運，並打開未來中央化的門。
- **要做**：①PyInstaller onedir + 安裝包（Inno/MSI）②程式碼簽章 ③版本顯示 + 更新提示（或靜默更新）④schema 驅動 migration + 完整測試 ⑤權限/角色（現場/內業/主管/管理員）⑥自我診斷頁 ⑦（若要進化）中央伺服器 + DB + API 的架構評估與 PoC。
- **不做**：一次到位的全功能伺服器（先 PoC 驗證再投入）。
- **驗收**：乾淨機器一鍵安裝即用；防毒不擋；舊資料升級不丟且可回滾並有測試；不同角色看到不同權限。
- **風險**：中央化是大工程，沒有真實多專案需求前不要硬上。
- **開發順序**：打包/簽章/安裝包 → 版本/更新 → migration 測試 → 權限 → （中央化 PoC）。

> **一句話原則**：**第 0-1 階段是「止血」，第 2 階段是「長肌肉」，第 3 階段是「能上戰場」。** 最常見的失敗是跳過止血直接長肌肉——加了一堆材料請款功能，結果雲端一同步全毀了。

<a name="十二三份結論"></a>
## 十二、三份結論

### 1. 給老闆看的版本（為什麼值得產品化）

我們已經有一個成熟的工具，能把現場修改單的拍照、焊口、產出 PDF 自動化，省下大量人工填表時間，而且程式品質遠高於一般內部小工具（已有完整測試與文件）。**但它目前只是「把單子印出來」——真正的金礦是那些被產出的資料本身。** 只要再投入一段時間把它升級成「修改單資料管理系統」，公司就能得到三件現在做不到的事：一是**每一筆請款都有材料與金額依據**，業主來對數時拿得出來、退件變少、收款更快；二是**所有修改單從建立到付款全程留痕可稽核**，誰做的、誰改的、請了沒、收了沒，一目了然，降低財務與糾紛風險；三是**材料用量與成本可統計**，知道每個專案的真實耗材成本。投入的重點不是炫炮功能，而是「資料不會掉、錢不會算錯、人能自己用」這些讓它敢被全公司每天使用的基礎。**這是一筆把『一次性省工時』變成『長期可累積的工程資料資產』的投資。**

### 2. 給工程師看的版本（下一步先改什麼）

別急著做材料和請款功能，先止血。**第 0 階段四件事，按順序：**（1）加單寫者鎖 + 心跳 + 唯讀模式——這是雲端共用不掉資料的根，現在 H4 是最大地雷。注意：`records.json` 和 `billing.json` 主檔其實**都已經是原子寫入了**（`.tmp`+`os.replace`，3 月稽核標的 M4 已被修掉），但原子只防單機斷電、**擋不住兩台電腦互相覆蓋**——你缺的是鎖；順手把附屬 JSON（dwg_map/weld_snapshot）也補原子、把 billing 納入 auto_backup（目前只備份 records）；（2）把「改焊口連動重命名」這種多檔操作包成 write-then-swap + journal，能 rollback；（3）把 settings.json 和 weld_snapshot.json 裡的絕對路徑（`C:\Users\a0976\...`）改成相對專案根，否則換機必死；（4）把 `_audit_data.py` 升級成內建的 `audit_integrity()`，啟動可選跑。同時順手清掉 Product_report_、tkinter_backup、debug_* 這些棄用碼，並把 main.py / gui.py 那段 ~200 行重複邏輯抽共用。**這些做完之前，不要碰新功能。** 材料系統記得：從一開始就照「方案 B 的 UI、方案 C 的 schema」設計，金額用 Decimal/整數分，價格存當下快照——這幾個決定錯了，半年後的資料就回不去了。

### 3. 給現場使用者看的版本（之後用起來會怎樣）

之後你打開工具，它會先確認你站在對的專案資料夾、資料是好的，再讓你進去——不會再開出一個空白畫面讓你猜發生什麼事。開新單會更快：拍照丟進收件匣，選日期、貼照片、點一句修改原因，三步就好，材料可以先不填、回辦公室再補。每張單長什麼狀態（還沒好、已產出、可以請款、已收款）會用顏色一眼看懂。最重要的是：**手滑改錯、刪錯東西，都救得回來；別人在另一台電腦開著同一份資料時，會提醒你現在是看的、不會兩個人改到打架。** 出錯的時候，畫面會用人話告訴你「發生什麼、現在按哪顆」，而不是跳一串看不懂的英文代碼；真的搞不定，按一顆「回報問題」就能把狀況打包寄給負責的人。**簡單講：會更快、更不怕按錯、更不會莫名其妙掉資料。**

---

> **結尾一句不客氣的話**：你現在最大的風險，不是「功能還不夠多」，而是「一個資料安全還停在單人等級的工具，正準備被丟到多人雲端環境裡經手公司的錢」。先把第 0 階段的止血做完，這個產品才真的站得住。其餘的材料、請款、打包，都是這個地基穩了之後的水到渠成。
