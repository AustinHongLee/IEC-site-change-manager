# 決策紀錄

此檔只記錄已收斂的產品與技術決策。發散討論請放在 `ai_sessions/` 或 `inbox/`。

## 已採納

### 2026-06-16：Opus 前導書升格為核心產品藍圖

- 決策：`00_核心文件_產品進化前導書_OPUS.md` 視為 Big Bang 產品化的核心原文。
- 原因：它完整涵蓋產品定位、UI/UX、資料模型、材料、請款、啟動守門、資料一致性、打包與測試。
- 影響：後續 AI 討論與開發規劃必須引用並對照此文件。

### 2026-06-16：第 0 階段優先處理資料安全

- 決策：先做啟動守門、單寫者鎖、備份、交易式寫入、一致性稽核。
- 原因：公司級使用最大的風險不是功能不足，而是多人雲端環境造成資料覆蓋、遺失或請款錯誤。
- 影響：材料系統、請款批次、新 UI、打包優化先不作為第一波主線。

### 2026-06-16：短期保留 JSON，不急著改 SQLite

- 決策：第 0 到第 1 階段仍以 JSON 為專案資料格式，但要加強原子寫入、備份、鎖與 schema 檢查。
- 原因：目前仍偏雲端資料夾共用，SQLite 放在同步資料夾有損壞風險。
- 影響：資料模型可以先以邏輯 schema 設計，但儲存層不急著改 DB。

### 2026-06-16：請款分項金額採自動計算加手動覆蓋

- 決策：請款面板載入時，若 `billing.json` 沒有手填焊口金額或材料金額，使用 `records.json` 的焊口明細與材料明細自動彙總；若已有手填分項值，優先保留手填值。
- 原因：材料與焊口金額應能追溯計算，但實務請款仍可能需要人工調整。
- 影響：分項自動計算值不直接寫死回 `billing.json`，除非使用者在請款面板雙擊分項金額欄位手動覆蓋。

### 2026-06-16：請款總金額不允許獨立手填

- 決策：`總金額` 永遠由 `焊口金額 + 材料金額` 計算，不再從 `billing.json` 的 `total` 欄位覆蓋。
- 原因：總額若可獨立手填，會造成分項金額與請款總額對不起來，後續追帳與稽核風險過高。
- 影響：舊資料若已有手填 `total`，請款面板會顯示差異警示；使用者儲存請款資料後，舊 `total` 覆蓋值會被清空，保留分項金額作為唯一依據。

### 2026-06-16：請款變更採 append-only 稽核紀錄

- 決策：每次請款面板儲存時，將 `billing.json` 的 old→new 差異寫入 `records/billing_audit.jsonl`；完全空白列不產生稽核雜訊。
- 原因：請款狀態、日期、分項金額與備註都屬於財務追蹤資料，不能只靠覆寫後的最後狀態。
- 影響：`operation_journal` 負責偵測中斷；`billing_audit.jsonl` 負責事後追查誰在何時改了哪一張單、哪些欄位從什麼值變成什麼值。

### 2026-06-16：請款狀態採 enum 與狀態機

- 決策：請款狀態集中定義於 `billing_status.py`，UI、材料鎖定與未來請款批次共用同一份允許狀態與轉換規則。
- 原因：狀態若是自由字串，會造成同義字、空白、跳狀態與重複請款風險，財務資料難以稽核。
- 影響：請款面板儲存前會拒絕非法狀態與不允許的直接跳轉；金額或狀態變更會先顯示摘要並要求二次確認，確認後才寫入 `billing.json` 與 `billing_audit.jsonl`。

### 2026-06-16：請款批次先做資料層與防重複

- 決策：新增 `records/billing_batches.json` 與 `billing_batch.py`，先支援批次建立、批次狀態轉換、活躍批次重複檢查；完整批次 UI 延後。
- 原因：真實請款是批次送出，不是一張單一格；同一張修改單若同時出現在兩個進行中批次，會造成重複請款或對帳錯亂。
- 影響：`草稿`、`請款中`、`已請款`、`部分付款`、`已付款`、`補件中`、`退回` 都視為活躍批次，只有 `已結案` 與 `作廢` 不再阻擋新批次；健康檢查會把重複活躍批次列為 error。

### 2026-06-16：請款批次 UI 先做建立入口

- 決策：請款追蹤面板先加入多選列與「建立批次」入口；建立前檢查未儲存變更、已結案/作廢單、活躍批次重複。
- 原因：完整批次管理 UI 會比較大，但公司使用時最需要先防止重複請款，並讓人能從現有請款表直接成批。
- 影響：活躍批次中的修改單會在請款表以 tooltip 提示；建立批次後先寫入 `billing_batches.json`，不自動改動 `billing.json` 狀態，避免隱性財務狀態變更。

### 2026-06-16：請款批次狀態管理先獨立於單據狀態

- 決策：請款追蹤面板加入批次清單與「更新狀態」入口，批次狀態使用 `billing_batch.py` 狀態機驗證。
- 原因：批次狀態與單張修改單的請款狀態雖然相關，但不能在未明確設計同步規則前互相自動覆寫，否則會產生隱性財務狀態變更。
- 影響：目前更新批次狀態只會寫入 `billing_batches.json`；每張修改單的 `billing.json` 狀態仍由請款表原本的二次確認與 audit 流程管理。

### 2026-06-16：請款金額採 TWD 整數進位與 5% 外加稅

- 決策：請款層金額使用 TWD，焊口金額、材料金額、未稅小計、稅額、含稅總額皆四捨五入到元；稅額為未稅小計 x 5% 外加。
- 原因：若不明定進位與稅制，多筆小數金額加總後會和人工發票、會計請款數字對不上。
- 影響：材料價目表與材料單價仍可保留小數精度；只有請款面板與請款匯出套用 `TWD_HALF_UP`，並在 `billing.json` meta 標記 `currency`、`tax_mode`、`tax_rate`、`rounding_rule`。

### 2026-06-16：材料價目表只做預設帶價，不回溯歷史

- 決策：每個專案使用 `records/material_pricebook.json` 作為材料預設單價來源；材料寫入修改單時，單價與金額拍照進 `records.json` 的材料明細。
- 原因：合約價或材料價可能變動，但既有修改單請款依據不可被未來價目表回頭改變。
- 影響：重跑產出時，已有單價的材料列會保留既有價格；價目表只補空白價格，不覆蓋手動或歷史快照。

### 2026-06-16：價目表先補來源、生效日與改價歷史

- 決策：價目表 item schema 升到 `1.1`，新增 `來源`、`生效日`；儲存價目表時保留 `history`，同一料號改價或改來源會追加 old→new 事件。
- 原因：只有 `價目表ID` 不足以追溯請款依據，因為料號價格會改；至少要能知道當時是合約價或報價，以及何時生效。
- 影響：材料帶價時會把 `價目來源`、`價目生效日` 拍照進材料明細；已存在的歷史價目依據不會被重跑產出覆蓋。

### 2026-06-16：材料類別先進 schema

- 決策：價目表與材料明細新增 `類別`，預設為「材料」，可選「材料 / 耗材 / 工資 / 雜項」。
- 原因：未來請款批次需要把材料、工資、耗材或雜項分線統計；若現在全部混成材料，日後 migration 會痛。
- 影響：現階段不強迫使用者細分；帶價時會把價目表類別拍照進材料明細，請款匯出仍沿用目前總額邏輯。

### 2026-06-16：材料配價失敗必須明示

- 決策：材料有零件資訊但價目表找不到對應單價時，不得把單價或金額靜默寫成 0；資料層標記 `配價狀態=missing_pricebook`，UI 顯示「未配價」並提示補價或手動輸入。
- 原因：靜默 0 元會讓請款少算且難以追查，比直接擋下或明示未配價更危險。
- 影響：未配價列不會覆蓋既有歷史價格；畫面提示文字不會被寫回資料層，使用者需補價目表或手動單價後才會成為可請款金額。

### 2026-06-16：材料受控詞彙集中管理

- 決策：新增 `material_constants.py` 作為材料零件、尺寸、SCH、材質與 alias 的共用入口，權威來源仍是 `wizard_data.json`；價目表、驗證器與 UI 編輯下拉都必須逐步收斂到此模組。
- 原因：材料配價是正規化字串比對，若 DeepSeek 骨架、UI 與執行期各自維護詞彙或 alias，會重新產生看不見的配價落空。
- 影響：舊資料的 `白鐵` / `SS` / `黑鐵` / `CS` 會正規化成完整材質；價目表骨架允許空白單價並以 `missing_price` 標示「有料號、未定價」，UI 顯示「未定價」而非 0 元。

### 2026-06-16：材料價目骨架匯入採 dry-run 與二次確認

- 決策：材料價目骨架匯入核心移到 `material_pricebook_importer.py`，CLI 與價目表 UI 共用同一套驗證、merge plan 與寫入規則；匯入時先顯示既有筆數、seed 筆數、將新增與已存在略過，確認後才寫入。
- 原因：價目表是請款依據，不能靠人工複製或工具自動覆蓋既有價目；同一 `(零件類型, 尺寸, SCH, 材質)` 已存在時必須跳過。
- 影響：`tools/import_pricebook_seed.py` 仍可供 AI/工程流程 dry-run；一般使用者可在材料價目表面板按「匯入骨架」完成同樣流程。

### 2026-06-16：補價後只重配未定價材料

- 決策：新增 `material_repricing.py`，價目表補價後可在 UI 套用到既有材料明細，但只處理未配價/未定價列；手動價與已請款鎖定修改單一律略過。
- 原因：補完價目表後，不應要求使用者重跑整個產出，也不能因補價動作回沖已請款或人工調整的財務資料。
- 影響：材料價目表 UI 可篩選「只看未定價」，空白單價以「未定價」提示；按「套用補價」前會先顯示可套用、仍未填單價、仍無價目、已請款略過與手動價略過數量。

### 2026-06-16：Opus Checkpoint B 回補重配稽核與未定價可見性

- 決策：材料重配價會寫入 `records/material_audit.jsonl` append-only 逐列稽核，並以 operation journal 包住備份、records 寫入與 audit append；單價或金額變動的修改單會標記 `需重產`。
- 原因：全檔備份不是可查詢稽核；補價後若已產出的 PDF/Excel 與 records 金額不同，使用者必須看見哪些單需要重產。
- 影響：紀錄管理面板新增「材料提醒」欄與「需重產」狀態；請款追蹤面板會標示含未定價材料的單，且建立請款批次時會擋下這些單。

### 2026-06-16：材料補價先支援篩選後批次填同價

- 決策：材料價目表由單選改為多選，新增「零件類型 / 材質」下拉篩選、「選取顯示」與「批次填價」；使用者可先篩選未定價，再把目前顯示列選取並填入同一單價、來源與生效日。
- 原因：匯入 442 筆骨架後，逐列雙擊填價過慢；但直接隱性批改全部篩選結果風險太高，因此採「先篩選、再選取、再確認、最後仍需按儲存」。
- 影響：批次填價只改目前選取的價目列，不直接寫檔、不套用到材料明細；仍須按「儲存」寫入價目表，再按「套用補價」更新未定價材料。

### 2026-06-16：合約價格表匯入只補新項與空價

- 決策：新增 Excel/CSV 材料價格表匯入；匯入前先驗證欄位、受控詞彙與單價格式，dry-run 摘要會列出新增、補空白單價、略過與價格衝突數量。
- 原因：公司實務更可能拿到整張合約價或報價表，不能要求逐筆手填；但既有有價項目若被表格覆蓋，會造成請款依據被無聲改寫。
- 影響：價格表匯入只新增新材料 key，或補既有骨架的空白單價；既有有價且匯入價不同時列為 conflict 並略過。寫入價目表後仍須另外按「套用補價」才會更新既有未定價材料明細。

### 2026-06-16：補價表可由系統匯出給 Excel 填寫

- 決策：材料價目表可匯出 CSV/XLSX 補價表，預設只輸出未定價項目；UI 匯出時優先使用選取列，其次使用目前篩選顯示列，最後才是全部價目。
- 原因：若公司沒有現成合約價表，使用者仍需要一份安全模板交給內部人員填價；模板欄位必須和匯入器一致，避免人工整理後欄位對不上。
- 影響：標準流程收斂為「匯入骨架 → 匯出補價表 → Excel 填單價 → 匯入價格表 → 套用補價」；匯出動作不寫入價目表、不改材料明細。

### 2026-06-16：需重產改為可操作工作隊列

- 決策：材料補價造成金額變更時，受影響修改單會標記 `需重產`；紀錄管理面板提供快速篩選與 CSV 匯出重產清單。重新產出成功寫回主紀錄時，自動清除舊的 `需重產` 旗標與原因。
- 原因：只顯示警告不足以支撐公司流程，使用者需要知道哪些單要排程重產，也不能在重產完成後繼續看到假警報。
- 影響：`需重產` 清單可交給現場或內勤安排重產；材料補價完成訊息會列出受影響報告編號，紀錄面板可匯出追蹤清單。

### 2026-06-16：現場資料核心固定，輸出模板可替換

- 決策：採納 `09_現場資料核心與多格式輸出前導書.md` 作為現場統計與多格式輸出的新主線；先建立 `CanonicalReport / CanonicalReportSet`，再談 Excel/PDF/照片表 renderer。
- 原因：每家公司格式不同，若把資料解析與輸出格式綁死，後續會變成每個客戶一段特例程式。固定資料核心後，輸出差異應落在 template mapping，而不是核心邏輯。
- 影響：新增 `canonical_report.py` 作為 records + attachments 的資料收斂層；renderer、現場統計單、PDF 拉框、Excel 貼圖都應逐步改吃這份模型。Excel COM 降級為 optional renderer，不得成為資料核心依賴。

### 2026-06-16：現場修改統計單先以標準 Excel 落地

- 決策：新增 `site_statistics_exporter.py`，以 `CanonicalReportSet` 輸出標準 `現場修改統計單.xlsx`，包含總覽、修改單清單、焊口統計、照片索引、照片表、用料統計、問題清單。
- 原因：現場階段最需要先有一張主管與內業都看得懂的統計單；先產標準統計單，比直接跳到 PDF 拉框或多公司模板更能驗證資料核心是否夠穩。
- 影響：紀錄管理面板新增「現場統計單」入口；這份統計單不依賴 Excel COM，不做報價與請款，只聚焦現場資料完整度、焊口、照片與用料。照片表會把 before/after 圖片直接嵌入 Excel，供「只需要照片貼入 Excel」的公司格式作為最小 renderer 基礎。

### 2026-06-17：Template Mapping 先做驗證閘門

- 決策：新增 `canonical_fields.py` 與 `template_mapping.py`，把 field-path catalog 從 `canonical_report.py` 拆成輕量契約，並提供 `validate_template.py` 驗證 `text / image / table` 三原語模板。
- 原因：多公司輸出格式不能靠每次改 renderer；模板引用欄位必須像材料受控詞彙一樣先驗證，避免 PDF/Excel 產出時靜默漏欄位。
- 影響：模板作者或 AI 交付前可用 `tools/validate_template.py` 檢查欄位錯字、table 欄位錯字、text/image 誤用集合路徑；欄位列表 CLI 不再載入整個 app log，輸出可直接作為模板作者文件。

### 2026-06-17：Renderer 前必須先 dry-run

- 決策：新增 `template_dry_run.py` 與 `tools/dry_run_template.py`，以 renderer-neutral 的資料結構預覽 template 對 CanonicalReport 的取值結果。
- 原因：現場輸出不能等到 Excel/PDF 產完才發現照片不存在、表格超列或欄位沒值；dry-run 要在產出前先把缺漏講清楚。
- 影響：目前可回報 text/image/table 欄位數、圖片路徑是否存在、表格列數與超出列數；後續 Excel/PDF renderer 與 UI 預覽都應吃這份 dry-run 結果。

### 2026-06-17：第一個 xlsx_template renderer 採純 openpyxl

- 決策：新增 `xlsx_template_renderer.py` 與 `tools/render_xlsx_template.py`，讓通過驗證的 `xlsx_template` 可實際把 `text / image / table` 寫入 Excel。
- 原因：多格式輸出需要先證明 template JSON 不是只會驗證，而是真的能產出；Excel 貼文字、貼圖、展表格是最小但最常見的公司格式需求。
- 影響：renderer 可使用既有 workbook 樣板或新建活頁簿；圖片不存在時會在目標儲存格留下提示並回報 warning；此路線不依賴 Excel COM。

### 2026-06-17：表格溢出不得寫過界

- 決策：採納 Opus Checkpoint C 的 C1 回補，`table` mapping 必須設定 `max_rows` 或 `rows_per_page`；dry-run 偵測到超出列數時升級為 error，renderer 不產出 workbook。
- 原因：若表格超出預留區仍照寫，會覆蓋下方既有儲存格，這比產出失敗更危險。
- 影響：`validate_template.py` 會擋下未設定列數上限的 table；`dry_run_template.py` 對 table overflow 回傳非 0；`render_xlsx_template.py` 不留下寫過界的錯檔。

### 2026-06-17：dry-run 必須回報模板未使用的現場資料

- 決策：採納 Opus Checkpoint C 的 C3 回補，dry-run 增加 `coverage.unmapped_data`，列出 CanonicalReport 中有值但模板未引用的欄位。
- 原因：現場統計與多格式輸出的風險不只是不小心引用錯欄位，也包括資料明明存在但某個公司模板沒有放進去。
- 影響：`dry_run_template.py` 可產出反向漏資料清單；未映射資料以 info 顯示，不阻擋產出。模板作者可用 `coverage_ignore` 明確忽略不需要輸出的欄位。

### 2026-06-17：xlsx renderer 產出後必須回讀校驗

- 決策：採納 Opus Checkpoint C 的 C2 回補，`xlsx_template_renderer` 產檔後重新讀取 workbook，校驗 text cell、table cell 與 image anchor。
- 原因：dry-run 只能預測資料與模板是否契合，不能證明檔案真的寫入成功；輸出後校驗是多格式 renderer 進入 PDF 前的必要安全網。
- 影響：render result 新增 `post_validation`；若輸出後校驗抓到 mismatch，結果會轉為 `ok=false` 並列出具體欄位/儲存格問題。

### 2026-06-17：xlsx template 需做 layout 碰撞檢查

- 決策：採納 Opus Checkpoint C 的 C4 xlsx 回補，`xlsx_template_renderer` render 前會檢查 text/image/table 佔用 cell 區域是否重疊或超出 Excel 邊界。
- 原因：圖片、表格與文字若互相覆蓋，即使資料寫入成功也會形成不可用報表；這類問題應在產檔前阻擋。
- 影響：render result 新增 `layout_validation`；模板圖片可用 `size_cells: [cols, rows]` 宣告佔用格數，未宣告時暫以 anchor cell 1 格估算。layout error 時不產出 workbook。

### 2026-06-17：照片 field-path 統一採 `[*]`

- 決策：採納 Opus Checkpoint C 的 C6 回補，`canonical_fields.py` 不再把 `photos.before[0..n]` / `photos.after[0..n]` 列為正式欄位，改以 `photos.before[*]` / `photos.after[*]` 作為 canonical 集合路徑。
- 原因：同一個集合若同時有 `[0..n]` 與 `[*]` 兩種正式寫法，模板作者與 AI 很容易混用，造成驗證、coverage 與文件說法不一致。
- 影響：`list-fields` 只輸出 `[*]`；舊模板若用 `[0..n]` 仍可被解析與驗證相容；`[0]`、`[1]` 等單張照片索引仍可用於 text/image。

### 2026-06-17：舊 Excel COM 產出先降級為可選能力

- 決策：採納 Opus Checkpoint C 的 C5 P0 回補，`excel_handler.py` 不再於 import 期載入 `win32com`；GUI/CLI 在啟動舊版修改單產出前先做 Excel COM 能力探測，不可用時以人話停用舊產出。
- 原因：公司級單一 exe 不能因未安裝 Office/pywin32 而連 App 都無法開啟；COM 應是舊模板後端，不是健康檢查、統計單、template validation/dry-run 的硬依賴。
- 影響：無 COM 環境仍可 import GUI 與核心模組；舊版「開始執行/重試失敗」會停用或提示原因；新現場統計單與 xlsx_template renderer 維持 COM-free。後續仍需 renderer registry 與舊報表 canonical 化。

### 2026-06-17：輸出後端改由 renderer registry 分派

- 決策：新增 `renderer_registry.py` 與 `tools/list_renderers.py`，把 `template.kind` 到 renderer backend 的關係集中管理；`render_xlsx_template.py` 改走 registry 分派。
- 原因：未來會同時有 `xlsx_template`、`pdf_overlay`、`photo_sheet`、`xlsx_com` 等後端，若每個 CLI/UI 自己寫 if，格式會再次爆炸。
- 影響：`xlsx_template` 是第一個正式 `CanonicalReport` renderer；`xlsx_com` 先登錄為 legacy optional backend，但標記 `CanonicalReport adapter pending`，不能透過 registry 繞過驗證直出。未來新增 renderer 需先進 registry，再接 UI。

### 2026-06-17：C5 先鎖 import-guard，GUI 讀 registry

- 決策：採納 Opus C5 Registry 再校準結果，暫不投資 `xlsx_com` CanonicalReport adapter；先新增正式 import-guard 測試，並讓 GUI 舊產出狀態讀 `renderer_registry`。
- 原因：`xlsx_com` 是要逐步淘汰的舊路，不應為它建立新的 canonical adapter；公司級部署更需要防止 COM 啟動期依賴復活，並讓 UI 的輸出能力來源集中。
- 影響：GUI 啟動時只讀 registry 的輕量 `unprobed` 狀態，不啟動 Excel；使用者按開始/重試時才完整 probe COM。正式測試會封鎖 `pythoncom/win32com` 並 import 核心與 GUI 集合，確保不碰 COM。

### 2026-06-17：非 COM PDF 路線先採 LibreOffice headless converter

- 決策：新增 LibreOffice capability probe 與 `workbook_pdf_converter.py`，用 `soffice --headless --convert-to pdf` 將已產出的 workbook 轉成 PDF。
- 原因：只要主要 PDF 交付物仍只能靠 Excel COM，無 Office 機器雖可啟動，仍無法完成公司常用輸出；LibreOffice converter 可先承接 `xlsx_template` 與現場統計單，不需重寫舊 COM 報表。
- 影響：新增 `tools/convert_workbook_pdf.py`；`tools/render_xlsx_template.py` 與 `tools/export_site_statistics.py` 可用 `--pdf-output` 在 workbook 成功後轉 PDF，紀錄管理面板也提供「統計PDF」入口。`settings.json` 可記住 `paths.soffice_path`，設定面板可瀏覽與測試 LibreOffice 路徑，並可執行輸出能力檢查。轉檔後會用 `pypdf` 回讀驗證頁數。LibreOffice 不可用時回人話錯誤，不影響 Excel workbook 產出。

### 2026-06-17：LibreOffice 部署策略採 portable 優先、settings fallback

- 決策：公司版非 COM PDF 的正式部署策略採 portable LibreOffice 隨打包發佈為主；`settings.json` 的 `paths.soffice_path` 作為 fallback；公司電腦已安裝 LibreOffice 時可容忍自動搜尋或指定路徑。
- 原因：只要求每台公司電腦自行安裝 LibreOffice 會遇到版本、字型、IT 權限與路徑差異；只靠 settings 則太依賴人工設定。portable 隨包最接近「單入口可用」的產品目標，settings fallback 可支援例外環境。
- 影響：在 portable 打包與真機測試落地前，PDF 不能被視為唯一交付物；所有 PDF 流程必須保留 xlsx 產出與人話降級提示。P0 真機驗收需包含 CJK 專案路徑、CJK 檔名、使用者已開 LibreOffice、缺 LibreOffice、timeout 與 PDF 可讀檢查。

### 2026-06-17：pdf_overlay 先進 schema-only，不先寫 renderer

- 決策：`pdf_overlay` 登錄進 renderer registry，但狀態固定為 `schema_only`；`validate_template.py` 可驗證 `report.v1 + template_mapping.v1 + pdf_overlay.v1`，正式 renderer 暫不開放。
- 原因：PDF overlay 必須共用 `text / image / table` 三原語與 field-path catalog；若在 renderer 前先發散成另一套 PDF 欄位、另一套 dry-run，就會破壞「現場資料核心固定、輸出模板可替換」的主線。
- 影響：PDF template 可先被 AI 或模板作者提交並驗證，檢查 `page`、`rect_norm`、座標越界、欄位重疊、table row limit、Excel target 欄位混用等錯誤；`render_with_template(kind="pdf_overlay")` 只回 `renderer_schema_only`，不產 PDF。

### 2026-06-17：輸出結果統一採 output_result.v1 envelope

- 決策：新增 `output_result.py`，主要輸出流程在既有細節欄位外，統一補 `result_schema_version`、`outputs`、`issues`、`capabilities`、`steps`。
- 原因：`xlsx_template`、LibreOffice PDF converter、現場統計單與未來 `pdf_overlay` 若各自回不同 JSON，UI、批次、AI 接管會重新變成特例判斷。
- 影響：既有詳細欄位如 `path`、`post_validation`、`pdf_conversion` 保留；新流程可優先讀 envelope。`render_with_template()`、`convert_workbook_to_pdf()`、`export_site_statistics.py --json`、`render_xlsx_template.py --json` 已開始輸出 `output_result.v1`。

### 2026-06-17：pdf_overlay renderer 先做最小垂直切片

- 決策：`pdf_overlay` 從 `schema_only` 升級為 `minimal` renderer；可將 `report.v1` 的 `text / image / table` 疊到 base PDF，並提供 `tools/render_pdf_overlay.py`。
- 原因：schema、output envelope、edge matrix 已就位，下一步需要用實際 PDF 疊圖驗證座標、圖片與表格輸出是否可行。
- 影響：registry 與輸出能力檢查會列出 `pdf_overlay` 可用；目前仍標記為 minimal，已補 CropBox、`/Rotate` regression、`text overflow=error` fail-fast 與 `table overflow=new_page`。`table overflow=truncate` 仍明確回 `overflow_mode_unsupported`，尚未宣稱支援多頁照片 grid、AcroForm 或真實公司表單驗證。

### 2026-06-16：已請款材料明細整列鎖定

- 決策：材料明細 upsert 主鍵必須包含 `SCH`，並且請款狀態進入 `已請款`、`請款中`、`已付款`、`部分付款`、`補件中`、`退回`、`已結案` 後，重跑產出不可新增或更新該修改單的材料列。
- 原因：不同 SCH 的同品名材料不可互相覆蓋；已送請款或已付款的單據不可因重跑產出、補價目表或數量變動而回溯改變請款依據。
- 影響：已請款資料若需修正，後續必須走人工解鎖、補件或調整單流程，不可由一般產出流程靜默修改。
