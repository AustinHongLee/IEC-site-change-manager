# AI 討論規則

這份規則是給後續參與本專案的 AI 使用，目標是讓討論能累積，而不是每次重頭開始。

## 進場前必讀

1. `README.md`
2. `00_核心文件_產品進化前導書_OPUS.md`
3. `02_決策紀錄.md`
4. `03_Phase0_落地開發板.md`

## 討論輸出格式

每次 AI 討論建議輸出到 `ai_sessions/`，檔名格式：

```text
YYYY-MM-DD_model_topic.md
```

例如：

```text
2026-06-16_opus_material-billing-review.md
2026-06-16_codex_phase0-implementation-review.md
```

## 每次討論應包含

- 本次角色：產品、UX、架構、QA、財務、現場、工程實作等。
- 參考文件：列出實際讀過的文件。
- 主要結論：最多 5 點。
- 風險與反對意見：必須寫，不可只寫贊成。
- 對 Phase 的影響：新增、調整、延後或拒絕。
- 需要人類決策的問題：清楚列出。

## 禁忌

- 不可把發散建議直接當成已決策。
- 不可跳過第 0 階段，直接做材料、請款或漂亮 UI。
- 不可假設目前程式已經有某功能；要回到 repo 驗證。
- 不可把「單一 exe」誤解成可以忽略 Excel COM、模板、設定、log 與資料夾結構。
- 不可自動刪除或覆蓋專案資料，除非已有明確備份與使用者確認。

## 收斂規則

任何提案要進入開發板，至少要回答：

- 解決什麼具體風險或使用者問題？
- 會改哪些資料模型或檔案？
- 有沒有 migration 或回滾需求？
- 驗收標準是什麼？
- 如果只做一半，會不會讓資料更危險？
