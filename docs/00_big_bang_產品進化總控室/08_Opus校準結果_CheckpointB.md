## 1. 已驗證可動的部分（實跑確認，給肯定）

我在沙箱用真實模組跑過，下列行為正確：

- **別名不再靜默 $0**：材料寫「白鐵」會正規化成「白鐵 (Stainless Steel)」並配到價(180×2=360)。Checkpoint A 的頭號地雷已解。
- **未定價 ≠ 0 元**：價目表有料、單價空白 → `配價狀態=missing_price`、金額空白；查無料 → `missing_pricebook`。兩者都不會算成 0。
- **重配安全邊界**：`is_reprice_candidate` 對 手動價→False、已有單價→False、missing_price→True。鎖定修改單(請款中/退回…)整張跳過。實跑 reprice plan：未鎖的補到價、鎖定的維持未定價、手動價原封不動。
- **狀態機**：`未請款→已付款` 被擋(不可跳)，`未請款→請款中` 放行。自由字串時代結束。
- **價目表 history**：改價/刪列會記 `price_changed`/`removed` 事件。
- **安全匯入**：dry-run 預設、`(零件,尺寸,SCH,材質)` 去重、不覆蓋既有價。
- **schema 補齊**：`類別`、`來源`(預設合約)、`生效日`、`價目表ID` 都進來了，補上 Checkpoint A 點的缺口。
- **配價 tie-break**：同分時優先選「有價」勝過「空價」的價目列(`material_pricebook.py:148-150`)，避免空價列遮蔽有價列。這個細節做得對。

## 2. 仍有缺口（依嚴重度）

### 🔴 G1. 重配沒有逐列 append-only 稽核
`apply_project_reprice_plan` = `auto_backup` + `_save_store`。全檔備份**不是**可查詢的稽核。
請款這條鏈需要的是「報告 R1 的材料 X 由『未定價』變成 75，動作=reprice，時間=T」這種逐列紀錄。
你已經有 `operation_journal`，接上即可。**這是請款批次之前最該補的一件。**

### 🟠 G2. 重配會改「未鎖定但已產出」修改單的金額，卻沒有產出物失效旗標
未請款/暫緩的修改單可以自由重配——財務上沒問題(還沒請)。但若這張單**已經產出 PDF/Excel**
交出去過(估驗報價)，補價後金額變了，產出物與 records 就**默默不一致**。
建議：重配後回傳「這次改了哪幾張修改單」的**清單**(不只總數)，並把這些單的產出物標記為「需重產」。
（這正是前導書 `source_fingerprint` 的用途。）對應你問的 4.1.4。

### 🟠 G3. 「鎖定」只在請款流程狀態觸發；已被人工核可但仍「未請款」的金額仍可能無聲變動
未請款的單重配自由，但若主管已用肉眼核過那個數字，它會在無提示下被改。
不是必修，但要靠 G2 的「改了哪些」清單 + 未定價總覽來補觀測性。

### 🟠 G4. build 與 apply 之間的 TOCTOU
`build_project_reprice_plan` 載入 store → 之後 `apply_project_reprice_plan` 整碗存回。
中間若有其他寫入會被蓋掉。有 Phase-0 單寫者鎖擋著，風險低；但財務寫入建議
「build+apply 同一把鎖內完成」或存檔前再確認 store 未變。

### 🟡 G5. 價目表 history 以 id 為鍵、且內嵌在價目表 JSON、每次存檔重建
id = `零件|尺寸|SCH|材質`。若某列改價同時又改了關鍵欄，會被當成 remove+add，歷史鏈斷。
而且 history 跟著 pricebook 一起被覆寫，不是獨立不可變帳。真要當稽核，history 應 append-only 寫到獨立檔。

### 🟡 G6. 仍無稅 / 未稅含稅 / 進位規則
Checkpoint A 就提過，到「請款批次」會變成**必補**：對業主請款一定要處理營業稅與整數進位，
否則逐列加總對不上發票。把它列為做請款批次的前置。

## 3. 回答你的校準問題

### 3.1 產品流程
1. **一個面板扛匯入+補價+套用補價，會不會混淆？** 會。`匯入骨架`是少用、近乎管理員的動作，
   `補價`/`套用補價`是日常。建議把`匯入骨架`收到「管理/設定」入口，日常面板只留補價流程，避免誤點重匯。
2. **`只看未定價`+`套用補價`順序直覺嗎？** 心智模型是 篩選未定價→在價目表填價→套用到材料明細。
   但「套用補價」其實動的是 **records 的材料明細**，不是畫面上的價目列——這是個情境跳躍。
   按鈕要講清楚它動什麼、動幾筆（見 3.3.2）。
3. **紀錄管理/請款追蹤要不要顯示未定價總數？** **強烈要。** 這是本輪最重要的安全網：
   防「以為入價了→請款少算」。建議 records 列上掛「未定價」徽章，且**請款追蹤對含未定價材料的單要硬擋或強警示**。
4. **補價後要自動提示哪些單金額變了，還是只給總數？** 給**清單**。只給總數會藏住「該回頭重檢/重產哪幾張」。與 G2 同源。

### 3.2 財務安全
1. **三條安全邊界(已請款鎖/手動略過/有價略過)夠嗎？** 邊界本身正確，但**少了可觀測性**：
   缺 G1(稽核)與 G2(產出失效)。邊界 OK，留痕不 OK。
2. **`missing_pricebook → missing_price → matched` 狀態流符合語意嗎？** 符合，而且**可操作**：
   `missing_pricebook`=系統不認得這料(要建料/匯入骨架)、`missing_price`=認得但沒定價(要補價)。
   兩者下一步動作不同，保留是對的。但 UI 要講「下一步要做什麼」，別只丟內部代碼名。
3. **重配未定價要不要 append-only audit？** 要。見 G1。
4. **材料明細本身的 old→new 要不要記？** 要，跟 G1 同一件事一起做。

### 3.3 UI/UX
1. **「未定價」清楚嗎？** 比會計術語清楚，保留。但要跟「查無價目」分開：建議
   「未定價（待補價）」與「查無價目（待建料）」，用動作語言。
2. **「套用補價」命名準嗎？** 不準，聽起來像在設定價。建議改「重算未定價材料（N 筆）」或「套用新價到未定價材料」。
3. **442 筆要更強篩選嗎？** 要：`零件類型`/`材質`/`尺寸段` 篩選 + 既有的`只看未定價`。單純子字串搜尋不夠。
4. **要不要批次填價 / 貼 Excel？** 要，而且是**最高槓桿**的 UX——合約價現實上就是 Excel 進來的。
   但必須走同一道 validation + 正規化閘門，不能繞過。

### 3.4 下一步排序（我的建議）

先做一個**小而關鍵的安全衝刺**，再進新功能：

- **第 0 步（小、先做）**：G1 重配 append-only 稽核 + G2/3.1.3 未定價跨面板可見性與「改了哪些單」清單。
  兩件都小，但請款批次會蓋在它們上面——別把高風險財務層蓋在「不可稽核、未定價看不見」的地基上。

接著依序：

1. **補價 UX**：批次填價 + 貼上/匯入 Excel 價格表(走閘門) + 未定價總覽。可用性最高、又直接消頭號風險。
2. **請款批次 UX**：批次、狀態同步、匯出。**但前置是先定 G6（稅/未稅含稅/進位）**，因為批次要算給業主的總額。
3. **財務 audit 深化**：把 G1 擴成完整的材料/請款變更稽核。
4. **單一 exe / 打包**：維持延後。

和 Codex 初判的差別：Codex 把 audit 放選項 3；我主張把「audit hook + 未定價可見性」抽出來當**第 0 步先做**，
成本低、且是請款批次的地基。

---

## 4. 一句話

方向對、落地紮實、紅旗有在收。進請款批次前，先補「重配留痕 + 未定價看得見」這兩塊小地基，
其餘照 3.4 排序走即可。
