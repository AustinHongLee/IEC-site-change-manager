# 現場資料核心與多格式輸出前導書

> 撰寫視角：產品架構師 × 文件渲染工程師 × 現場工務主管 × 內部工具導入顧問
> 一句話定位：**現場資料核心固定，輸出模板可替換。**
> 對應前一階段：材料/請款線（00、05～08）。本書把主線拉回「現場資料 → 完整統計單 → 各公司格式輸出」。

---

## 0. 核心主張與最大風險

這個工具真正成熟、別人難取代的核心是**現場階段**：紀錄修改、統計焊口、整理 before/after、紀錄用料、產出各種單。
所以產品的骨幹不該是「我們支援哪幾種匯出」，而是：

> **一份固定、可驗證的內部資料模型（Canonical Report），加上一層可替換的模板映射（Template Mapping），讓每家公司的格式只是一份 JSON 模板，不是一段程式碼。**

最大風險不是「格式不夠多」，而是三件事，本書每一章都在防它們：

1. **格式爆炸**：每來一家公司就改一次核心、加一個 if，最後沒人敢動 renderer。
2. **工務人員被迫當設計師**：把「拉框、定義映射」這種高階工作丟給現場，結果沒人會用。
3. **COM 綁死核心**：核心資料、統計、驗證都依賴 Excel COM，部署一不穩整個工具就死。

對應的三個設計決策（全書反覆出現）：

- 用**封閉的欄位路徑目錄（field-path catalog）**當模板與核心之間的唯一契約。模板只能引用目錄裡的路徑；引用不存在的路徑＝驗證失敗。這跟材料線的「受控詞彙閘門」是同一招。
- 把角色拆成**模板作者（office/Codex/AI，一家公司做一次）** 與 **模板使用者（現場/內業，選模板、按產出）**。設計器是作者工具，不是日常工具。
- COM **移出核心**，降級成 optional renderer；核心只認 Canonical Report 與 openpyxl／PDF 函式庫。

## 1. Canonical Data Model（CanonicalReport）

### 1.1 設計原則

- **單一真實來源**：所有 renderer 只吃 CanonicalReport，不直接讀資料夾、不各自解析 `materials.txt`。
- **欄位穩定、輸出多變**：模型欄位增刪要版本化（`schema_version`）；輸出模板永遠對著版本化的欄位路徑。
- **衍生值預先算好**：`welds.summary`、`materials.summary`、`photos.before[0]` 這類「模板會用」的彙總與索引，由核心算好放進模型，模板不做運算。模板只做「把某個路徑的值貼到某個位置」。
- **完整度是模型的屬性，不是模板的責任**：缺照片/缺 note/缺材料的判斷屬於 CanonicalReport，與要輸出成哪種格式無關。

### 1.2 模型結構（對齊目前實際資料）

目前現場資料散在：`records.json`（records/details/materials）、附件資料夾（`weld_info.json`、`GroupWeld.txt`、`note.txt`、`materials.txt`、`before*.jpg`/`after*.jpg`、附件 PDF）、`內容指紋`。CanonicalReport 是把這些收斂成一份結構：

```jsonc
{
  "schema_version": "report.v1",
  "report": {
    "report_id": "…",        // 報告編號
    "date": "2026-01-12",    // 日期
    "series": "547",         // Series NO
    "dwg_no": "…",           // DWG NO
    "change_type": "裁切重焊", // 變更類型（加長/裁切重焊…）
    "description": "…",       // 說明 / 修改原因敘述
    "folder": "0547_AG",      // 資料夾名（provenance）
    "fingerprint": "md5…"     // 內容指紋（產出當下來源指紋）
  },
  "welds": {
    "rows": [
      { "weld_no": "1", "mark": "r", "size": 2.0, "material": "", "thickness": "", "code": "1r2", "is_cut": true }
    ],
    "count": 7,
    "total_size": 13.0,
    "summary": "1r2、2r2、3r2…（共7口）",   // 預先算好的人話彙總
    "by_size": [ { "size": 2.0, "count": 5 } ]
  },
  "materials": {
    "rows": [
      { "component": "Elbow (彎頭)", "size": "2\"", "sch": "", "material": "白鐵 (Stainless Steel)",
        "qty": "1", "unit": "個", "category": "材料", "remark": "" }
    ],
    "count": 3,
    "summary": "彎頭×1、三通×1、法蘭×1",
    "by_component": [ { "component": "Elbow (彎頭)", "qty": 1 } ]
  },
  "photos": {
    "before": [ { "path": "…/before.jpg", "exists": true, "w": 4032, "h": 3024 } ],
    "after":  [ { "path": "…/after.jpg",  "exists": true, "w": 4032, "h": 3024 } ],
    "mode": "single",           // single | dual（before_1/2、after_1/2）
    "has_before": true,
    "has_after": true
  },
  "attachment_pdf": { "path": "…/547.pdf", "name": "547.pdf", "exists": true, "pages": 1 },
  "note": { "raw": "…", "is_placeholder": true },   // note.txt；偵測「# 請填寫…」樣板＝尚未填
  "completeness": {
    "level": "incomplete",      // complete | incomplete | empty
    "missing": ["note", "after_photo"],
    "flags": { "note_is_placeholder": true, "no_material": false }
  },
  "provenance": {
    "source_files": ["weld_info.json","GroupWeld.txt","note.txt","materials.txt"],
    "collected_at": "2026-06-16T…"
  }
}
```

對「現場統計單」這種多筆彙總，定義一個更上層的 **CanonicalReportSet**：`{ project, reports: [CanonicalReport…], aggregates: {…}, issues: […] }`，第 6 章用它。

### 1.3 完整度檢查（模型內建，與輸出無關）

每份 report 算出 `completeness`：`empty`（幾乎沒資料）/`incomplete`（缺關鍵項）/`complete`。關鍵項至少含：有焊口或有材料、note 非樣板、before/after 視案場要求、附件 PDF（若該案場需要）。
這份結果同時餵給 UI（紅黃綠燈）與第 8 章的 dry-run。**重點：完整度不看模板**，避免「換個模板完整度就變了」這種荒謬。

## 2. Template Mapping Layer

### 2.1 只用三種映射原語（防止格式爆炸的關鍵）

不論哪家公司、Excel 還是 PDF，所有映射都收斂成三種原語。renderer 只要實作這三種，就能渲染任意模板：

| 原語 | 來源 | 目的地 | 一對多？ |
|---|---|---|---|
| `text` | 一個 field-path 的純值（或彙總） | Excel cell／PDF rectangle／PDF freetext | 否 |
| `image` | 一張照片 path | Excel cell anchor／PDF rectangle | 否（但可被 `table` 包成多張） |
| `table` | 一個 list field-path（welds/materials/photos） | Excel 起始列＋欄位對應／PDF 重複區塊 | 是（自動分頁） |

「重複區塊」（多焊口、多照片、多材料）**一律是 `table`**，帶一個 row-template。這樣「一對多」只有一種處理路徑，不會每種格式各寫一套。

### 2.2 欄位路徑目錄（field-path catalog）— 模板與核心的唯一契約

模板只能引用目錄裡的路徑。這是「AI 只新增模板、不改核心」的命脈：

```
report.report_id    report.date       report.series      report.dwg_no
report.change_type  report.description report.folder
welds.summary       welds.count       welds.total_size
welds.rows[*].weld_no  welds.rows[*].code  welds.rows[*].size  welds.rows[*].mark
materials.summary   materials.count
materials.rows[*].component  materials.rows[*].size  materials.rows[*].sch
materials.rows[*].material   materials.rows[*].qty   materials.rows[*].unit
photos.before[*] photos.after[*]  photos.mode
attachment_pdf.path note.raw
```

- `[*]` 是正式的集合表示法，適合 `table` / grid；`[0..n]` 只留在文件文字說明或舊模板相容，不由 `list-fields` 匯出。
- `[0]`、`[1]` 等定值索引可用在 `text`/`image`，例如 `photos.before[0]` 或 `photos.before[0].path`。
- 核心發佈這份目錄（程式可 `list-fields` 匯出，跟 `validate_pricebook --list-vocab` 同概念）。
- 模板引用未知路徑 → 驗證直接擋下（第 8 章）。**新增公司格式 = 新增一份模板 JSON，零核心改動。**

### 2.3 一對多的三種落地

- **Excel**：`table` 指定起始 cell + 欄位→欄母對應，往下展開；超出預留列數 → 自動插列或換頁（第 8 章）。
- **PDF 重複區塊**：`table` 指定一個 row 矩形 + 列高 + 每頁列數，往下堆疊；滿頁 → 續頁。
- **多頁照片表**：`photos.before[*]`/`after[*]` 當成 `table`，每頁 N 格（grid），照片數超過 → 自動加照片頁。

## 3. PDF 模板設計器概念

### 3.1 角色定位（吹毛求疵：拉框不能丟給現場）

PDF 拉框是**模板作者**的事（office/power user/Codex/AI 做一次），不是現場日常。設計器產出一份 `template.json`，現場使用者只會「選這份模板 → 產出」。本書把設計器定義成一個「在底圖上拉框、命名、存 JSON」的小工具，但**它不在現場主流程裡**。

### 3.2 命名規則（直接用你提的 path，並加可讀後綴）

框名 = `field-path`，可加 `@用途` 後綴方便人看：

- 文字框：`report.report_id`、`report.date`、`welds.summary`、`materials.summary`
- 圖片框：`photos.before[0]`、`photos.after[0]`
- 表格區：`@table:welds`、`@table:materials`、`@grid:photos.before`
- 重複列原型：表格區內再放子框 `welds.rows[*].code`、`welds.rows[*].size`（相對座標）

命名必須能被核心目錄驗證；打錯字＝存檔時就紅字擋下，不會等到產出才靜默漏掉（呼應材料線「靜默失敗」教訓）。

### 3.3 框的型別

| 型別 | 內容 | 溢出策略 |
|---|---|---|
| 文字框 text | 單一路徑值 | 縮字→換行→截斷標記（第 8 章）|
| 圖片框 image | 單張照片 | 等比縮放置中，留白；可設定 `fit: contain/cover` |
| 表格區 table | list + 列原型 + 每頁列數 | 滿頁續頁 |
| 重複格 grid | 照片陣列 | 滿格加頁 |

### 3.4 座標系統（吹毛求疵：最容易爆的地方）

PDF 原點在左下、單位是 point（72/吋）；多數設計 UI 是左上、像素。**模板必須存「正規化、可見頁面相對」座標**，renderer 再依實際 PDF 頁面換算。否則換一張不同 DPI/尺寸的底圖，所有框就跑掉。

- 每個框存：`page`、`rect_norm: [x,y,width,height]`。
- `page` 從 1 開始，與使用者看到的 PDF 頁碼一致。
- `rect_norm` 全部為 0~1，相對 PDF 的可見區（CropBox）；若 PDF 沒有 CropBox，renderer 才退回 MediaBox。
- `rect_norm` 原點定義在可見區左上角，x 向右、y 向下；renderer 內部再轉成 PDF 左下座標。
- renderer 必須以「已套用 `/Rotate` 後的可見幾何」換算座標，避免 90/180/270 度旋轉頁貼歪。
- `page_size_pt` 可作設計工具的提示或除錯資訊，但不得作為 `pdf_overlay.v1` 的座標基準。
- 紅框/框線只是設計輔助：`"debug": true`，正式輸出預設隱藏。

### 3.5 mapping JSON 範例（PDF overlay）

```json
{
  "template_id": "acme_pdf_v1",
  "schema_version": "template_mapping.v1",
  "target_schema_version": "pdf_overlay.v1",
  "kind": "pdf_overlay",
  "base_pdf": "templates/acme/base.pdf",
  "coordinate_space": "normalized",
  "defaults": { "font": "NotoSansTC", "size": 11, "overflow": "shrink" },
  "fields": [
    { "type": "text",  "source": "report.report_id", "page": 1, "rect_norm": [0.72,0.06,0.23,0.04], "overflow": "shrink" },
    { "type": "text",  "source": "report.date",      "page": 1, "rect_norm": [0.72,0.11,0.23,0.04], "overflow": "shrink" },
    { "type": "text",  "source": "report.description","page": 1, "rect_norm": [0.08,0.20,0.87,0.14], "overflow": "wrap" },
    { "type": "image", "source": "photos.before[0].path", "page": 1, "rect_norm": [0.08,0.40,0.40,0.30], "fit": "contain" },
    { "type": "image", "source": "photos.after[0].path",  "page": 1, "rect_norm": [0.52,0.40,0.40,0.30], "fit": "contain" },
    { "type": "table", "source": "welds.rows", "page": 1,
      "rect_norm": [0.08,0.72,0.84,0.23], "rows_per_page": 8, "overflow": "new_page",
      "columns": [
        { "source": "code", "width_norm": 0.25 },
        { "source": "size", "width_norm": 0.25 },
        { "source": "mark", "width_norm": 0.25 },
        { "source": "weld_no", "width_norm": 0.25 }
      ]
    }
  ]
}
```

## 4. Excel 模板設計器概念 & COM 隔離

### 4.1 現況與問題

目前 `excel_handler.py` 用 `win32com` 開 Excel、跑 macro 貼圖（`get_template_for_mode` 給 `before_ranges/after_ranges`）、`ExportAsFixedFormat` 轉 PDF。問題你列得很準：要裝 Excel、部署不穩、轉 PDF 受環境影響、背景程序殘留、公司級 exe 難維護。**好消息：核心資料（records.json、openpyxl 匯出）本來就不碰 COM，COM 已侷限在 `excel_handler.py`。** 我們要做的是把這道牆砌死。

### 4.2 不依賴 COM 的 Excel 路線（預設）

- **文字填值**：openpyxl 直接寫 cell。模板 JSON 用 `text` 原語對 `sheet!cell`。
- **貼照片到 Excel**：openpyxl 的 `Image` + cell anchor **就能嵌圖，不需要 COM**。「只把 before/after 貼進 Excel」這種最常見需求，純 openpyxl 即可。
- **表格分頁**：openpyxl 寫列、預留樣式列；超量插列或開新 sheet。
- **Excel → PDF**：預設用 **LibreOffice headless**（`soffice --headless --convert-to pdf`）轉檔，跨機穩定、無殘留程序、可進 exe 隨附。COM 不是預設轉檔器。

### 4.3 Excel 模板 JSON 範例（openpyxl 路線）

```json
{
  "template_id": "bclient_xlsx_v1",
  "schema_version": "report.v1",
  "kind": "xlsx_template",
  "workbook": "templates/bclient/base.xlsx",
  "sheet": "修改單",
  "fields": [
    { "type": "text",  "source": "report.report_id", "cell": "G2" },
    { "type": "text",  "source": "report.date",       "cell": "G3" },
    { "type": "text",  "source": "report.description", "cell": "B5", "wrap": true },
    { "type": "image", "source": "photos.before[0]", "anchor": "B10", "size_cells": [6, 10] },
    { "type": "image", "source": "photos.after[0]",  "anchor": "I10", "size_cells": [6, 10] },
    { "type": "table", "source": "materials.rows", "start_cell": "B24",
      "columns": ["component","size","sch","material","qty","unit"],
      "max_rows": 12, "overflow": "new_sheet" }
  ]
}
```

### 4.4 COM 隔離成 optional renderer（吹毛求疵：該不該完全移出核心？）

**該。核心邏輯（資料、統計、驗證、預設 Excel/PDF）絕不 import `win32com`。** COM 只保留為「進階保真渲染器」的可選插件，用於：(a) 既有 macro 模板的相容、(b) 對版面保真要求極高、LibreOffice 轉檔不夠像時。規則：

- COM renderer 放獨立模組，`import win32com` 只在這裡。
- 啟動時做能力偵測（`capabilities.excel_com = True/False`）；沒有就灰掉這個輸出選項，不讓核心崩。
- 任何核心測試、`--health-check` 都能在無 Excel 環境跑綠。
- COM 失敗一律降級到 openpyxl + LibreOffice，並在輸出報告標記「以相容模式產出」。

## 5. Renderer Architecture

單向資料流，元件各司其職、可單測：

```
資料夾/records.json
      │  DataCollector        # 唯一讀檔解析處；產出 CanonicalReport(Set)
      ▼
CanonicalReport(Set)         # 不可變、版本化、含 completeness
      │  TemplateDefinition   # 載入 template.json（pdf_overlay / xlsx_template / photo_sheet）
      ▼
MappingResolver              # 把 field-path 解析成實值/清單；驗證路徑存在；算分頁；不畫圖
      │  → RenderPlan（每個目的地要放什麼、放哪、預期幾頁）
      ▼  分派到對應 renderer
ExcelRenderer  PdfOverlayRenderer  PhotoSheetRenderer   # 只負責把 RenderPlan 畫出來
      ▼
輸出檔 + RenderResult
      │  ValidationReporter   # 合併 dry-run(前) 與 render(後) 結果成校驗報告
      ▼
產出紀錄（journal）
```

元件契約（介面草圖）：

- `DataCollector.collect(folder|store) -> CanonicalReport(Set)`：唯一碰原始檔的地方。
- `TemplateDefinition.load(path) -> template`：schema 與 field-path 靜態檢查。
- `MappingResolver.resolve(report, template) -> RenderPlan`：把路徑變實值、決定分頁、回報「未映射欄位」「找不到的路徑」。**不依賴任何渲染庫**，純資料，最好測。
- `Renderer.render(plan) -> RenderResult`：每種輸出一個，介面一致（`supports(kind)`、`render(plan)`）。
- `ValidationReporter.report(dry_run, render_result) -> ValidationReport`：人話化（缺照片、3 筆材料未進表、說明縮字、第 4 頁為續頁…）。

新增一家公司格式：丟一份 `template.json` →（必要時）對應 renderer 已存在 → 完成。要新 renderer 才動程式碼，而三種原語已涵蓋絕大多數需求。

## 6. 現場修改統計單（標準內容）

用 CanonicalReportSet 產出，一份標準統計單應含以下 section（Excel 多 sheet 或 PDF 多段）：

| Section | 內容 | 來源 |
|---|---|---|
| 總覽 | 案場、期間、修改單數、焊口總數/總尺寸、材料筆數、完整度統計（幾張缺照片/缺note） | aggregates |
| 修改單清單 | 每張：報告編號、日期、Series、DWG、變更類型、焊口摘要、完整度燈、縮圖 | reports[] |
| 焊口統計 | 依尺寸/類型彙總、總口數、總尺寸 | welds aggregate |
| 照片索引 | 每張單的 before/after 縮圖與檔名、缺哪張 | photos |
| 用料統計 | 依 零件/尺寸/SCH/材質 彙總數量（沿用現有彙總邏輯） | materials aggregate |
| 問題清單 | 跨單彙整：缺照片、缺 note、缺材料、未產出、需重產（指紋變了） | completeness + provenance |

「問題清單」是這份統計單的靈魂——它把「現場到底還缺什麼」一次攤開，正是工務主管最需要的。「需重產」用 `report.fingerprint` 與上次產出時的指紋比對得出。

## 7. UI / UX（給一般工務人員）

主流程刻意做成一條直線，**不出現任何「拉框/映射/路徑」字眼**：

1. **選案場資料夾** → 自動掃描、建 CanonicalReportSet。
2. **檢查完整度** → 紅黃綠清單：哪幾張缺照片/缺 note/缺材料/未產出。可直接跳到該資料夾。
3. **選輸出模板** → 從已安裝模板清單挑（「A 公司 Excel」「B 業主 PDF」「照片表」）。模板由作者預先備好。
4. **預覽缺漏（dry-run）** → 在產出前就說：「3 筆材料放不下會分頁」「2 張單沒有 after 照片」「說明過長會縮字」。
5. **產出** → Excel / PDF / 照片表，預設 openpyxl + LibreOffice，無需 Excel。
6. **顯示哪些資料沒填進模板** → 輸出後校驗報告：哪些欄位模板沒用到、哪些資料超出版面。
7. **留下產出紀錄** → 寫 journal：誰、何時、用哪個模板版本、產出哪些檔、來源指紋。

角色分離再強調一次：**現場/內業只走 1–7；模板作者另有一個「模板管理/設計器」入口**，不混在主流程。

## 8. 安全與驗證

兩類驗證要分開，dry-run 兩個都跑：

- **資料完整度**（與模板無關）：缺照片/缺 note（含偵測樣板字）/缺材料/未產出。
- **渲染契合度**（report × template）：下列情境。

| 情境 | 處理 |
|---|---|
| 模板引用的欄位路徑不存在 | **載入/存模板時就擋**（靜態檢查比對 field-path 目錄），錯字不放行。 |
| 照片數量超出框位 | 多的轉到照片表附頁，或標「超出 N 張未列入」；絕不靜默丟。 |
| 材料/焊口列超出表格區 | 自動分頁（Excel 插列或新 sheet；PDF 續頁），頁尾標「續下頁」。 |
| PDF 文字溢出 | 縮字 → 換行 → 仍不行則截斷並標記 `…(溢出)`，列進校驗報告。 |
| 路徑有值但對應照片檔不存在 | 框留白 + 報告標「photos.after[0] 檔案遺失」。 |
| 輸出前 dry-run | 不產檔，回報：將產幾頁、哪些欄位未映射、哪些資料放不下、缺哪些料/照片。 |
| 輸出後校驗報告 | 實際產出後比對：每個 field-path 是否成功放入、分了幾頁、哪些被截斷/超出。 |

原則：**寧可大聲報「沒放進去」，絕不安靜地漏**。這是材料線「靜默 $0」教訓直接搬過來。

## 9. 分階段落地計畫

### Phase 0：資料核心歸一（不碰新輸出）
- **目標**：DataCollector + CanonicalReport(Set) + 完整度檢查，成為唯一資料來源。
- **範圍**：把現有散落解析（weld_info/GroupWeld/note/materials/photos/附件/指紋）收斂進 collector；發佈 field-path 目錄與 `list-fields`。
- **不做**：模板設計器、PDF overlay、LibreOffice 轉檔。
- **驗收**：既有 Excel 輸出改為「先建 CanonicalReport，再餵舊輸出」，產出結果與現況一致（回歸測試）；無 Excel 環境也能建出完整 CanonicalReport 並跑完整度檢查。
- **風險**：collector 與舊解析行為不一致 → 用既有資料夾做 golden-file 比對。

### Phase 1：Mapping 層 + openpyxl renderer + dry-run（最有感、風險中）
- **目標**：`text/image/table` 三原語 + MappingResolver + ExcelRenderer（純 openpyxl，含貼照片）+ dry-run/校驗報告。
- **範圍**：xlsx_template JSON、未映射/超量報告、UI 第 1–7 步雛形（模板用手寫 JSON）。
- **不做**：PDF overlay、設計器 GUI、COM。
- **驗收**：用 2 家不同 Excel 格式，各一份模板 JSON，零核心改動產出；dry-run 能預告分頁與缺漏。
- **風險**：照片 anchor 尺寸/比例在不同 Excel 版本顯示差異 → 固定 anchor + 等比縮放，文件註明。

### Phase 2：PDF overlay renderer + 照片表 + LibreOffice 轉檔（風險較高）
- **目標**：PdfOverlayRenderer（固定底圖貼字/圖/表+續頁）、PhotoSheetRenderer、Excel→PDF 走 LibreOffice。
- **範圍**：pdf_overlay JSON、正規化座標換算、溢出策略、多頁照片表。
- **不做**：拉框 GUI 設計器（模板先由 Codex/AI 手寫或半自動產）。
- **驗收**：一張固定底圖 PDF + 一份模板，能把文字/焊口表/before-after 貼到指定範圍並正確分頁；LibreOffice 在乾淨環境轉 PDF 成功。
- **風險**：座標/字型/中文字嵌入 → 內嵌 Noto 字型、座標用正規化+頁尺寸、加產出後校驗。

### Phase 3：PDF/Excel 模板設計器 GUI + COM optional renderer（最後做）
- **目標**：作者在底圖上拉框命名存 JSON；COM 降級為可選保真渲染器。
- **範圍**：設計器（框型別、命名驗證、紅框 debug 隱藏、即時對 field-path 目錄校驗）、COM 隔離模組 + 能力偵測 + 降級。
- **不做**：讓現場使用設計器（它是作者工具）。
- **驗收**：非工程作者能在 30 分鐘內為一家新公司做出可產出的模板，全程不改程式碼；移除/缺少 Excel 時核心與所有非 COM 輸出仍正常。
- **風險**：設計器複雜度反噬 → 嚴守三原語，不為單一公司加特例型別。

## 10. 吹毛求疵自檢（你點名的六問）

1. **mapping 會不會過度複雜？** 會，如果放任每家公司加新原語。對策：**鎖死三原語（text/image/table）**，重複區塊一律走 table。複雜度被天花板封住。
2. **一般工務人員能操作嗎？** 能，只要**把「拉框/映射」從主流程移除**。現場只做選資料夾→看完整度→選模板→產出。設計是作者的事。
3. **PDF 拉框命名實用嗎？** 實用，但前提是**框名＝受控的 field-path、存檔即驗證**。自由命名會重演材料線的靜默失敗；命名綁目錄才安全。
4. **Excel COM 該完全移出核心嗎？** 該。核心永不 import COM；COM 只當 optional 保真渲染器，預設用 openpyxl + LibreOffice。核心測試與 health-check 必須在無 Excel 環境全綠。
5. **如何避免格式讓程式碼爆炸？** 唯一契約＝field-path 目錄 + 三原語 + 模板 JSON。新增公司＝新增 JSON，不改核心；要動核心的只有「新原語/新 renderer」，而那極少發生。
6. **如何讓 AI 只新增模板、不改核心？** 給 AI 的任務鎖成「產一份 template.json，引用目錄內路徑，通過 `validate_template`」——跟材料線「產 seed、過驗證閘門」同一套。核心不因模板而變。

- 建議新增工具：`tools/validate_template.py`（對 field-path 目錄靜態驗證模板，跟 `validate_pricebook.py` 對齊）、`python -m render list-fields`（匯出目錄）。

## 附錄 B：給模板作者 / AI 的鐵則（雛形）

1. 模板只能引用 `list-fields` 目錄內的路徑；未知路徑＝不合格。
2. 一對多只能用 `table`，要給 `overflow` 策略。
3. PDF 座標一律用 `rect_norm: [x,y,width,height]`，相對 CropBox 可見區，page 從 1 開始；紅框設 `debug:true`。
4. 不得在模板內做運算或硬寫值；值來自 CanonicalReport。
5. 交付前過 `validate_template`，0 ERROR 才算完成。
