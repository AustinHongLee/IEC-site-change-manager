# 技術路線決策（v0.1）— 工務修改單

> **這份在幹嘛**：記錄「UI 用什麼技術、資料/部署往哪走」的決策與理由，讓任何接手的人（未來的你 / Codex / 同事）有依據，不必重吵一遍。地位等同 schema 那份。

---

## 1. 兩條軸，先分清楚（最重要）

| 軸 | 現在 | 中期 | 未來 |
|---|---|---|---|
| **UI 技術** | PyQt6 | HTML/JS（pywebview 原生殼） | （可能）瀏覽器 |
| **資料 / 部署** | 本機 / 共用資料夾 | 共用資料夾 + 檔案鎖 | 中央伺服器 + DB |

- **「給公司用」壓最重的是「資料/部署軸」**（多人寫、更新、IT），不是 UI 軸。
- UI 軸是下游，而且因為引擎乾淨 → **低後悔**。
- 兩軸要分開決策，別混在一起吵。

---

## 2. 不變的地基（資產，任何決策都不准動它）

`change_order`（canonical 記錄）＋ `weld_lookup` ＋ `weld_codec` ＋ `change_order_builder` ＋ `change_order_store`：
全 Python、headless、不認 UI、有測試。

**這層是真資產**：換 UI、換成伺服器後端，它都不用重寫。
**鐵律：任何決策都不准把邏輯塞回 UI。**

---

## 3. 決策

### D1 — UI 走 HTML/JS，近期用 pywebview 當原生桌面殼
- **為什麼**：公司用 → 可能中央化；**HTML/JS 前端「可帶著走」**（pywebview 桌面 → 以後瀏覽器/中央 API 幾乎不用重畫），**Qt 是丟棄式**（走到網頁那天整個重做）。
- pywebview 前後端是**原生橋（非 HTTP）**：JS `pywebview.api.x()` 直呼 Python，無 server / port / CORS → 符合「不要瀏覽器的感覺」。
- **對本案的 cons 校準**：跨平台跑版（Windows 單平台 → 不存在）、呼叫延遲（表單型 → 無感）、DevTools（WebView2 ＝ 完整 Chromium）。**真正剩的風險 ＝ 打包**（見 §5 spike）。

### D2 — 引擎保持「可被本機呼叫、也可被伺服器包起來」
- 現在：pywebview 的 `js_api` 直接包 `builder` / `store`。
- 未來：同一批引擎方法改用 FastAPI 之類暴露成 HTTP API，前端把 `pywebview.api.x()` 換成 `fetch(...)` → 中央版。
- **現在別把「本機路徑 / 單機假設」寫進 UI**，把這條路留著。

### D3 — 中央伺服器 + DB 是「規劃中的未來」，現在不建、但不擋死
- 現在最該止血的是**多人寫共用資料夾的鎖**（記憶裡的地雷 #1）——這跟 UI 無關，且比 UI 漂不漂亮更緊急。
- 不現在建伺服器（太早），但設計上**不擋它**（引擎已 ready、UI 走 HTML 已 ready）。

---

## 4. 現在做 / 延後 / 別擋死

**現在做**
1. **半天 packaging spike**：`pywebview + pyinstaller + WebView2` 丟到**實際公司機器**驗能不能裝、能不能開（決定全押前先驗，見 §5）。
2. 新精靈 UI 照「新精靈完整規劃 v0.1」用 **HTML 重做**（引擎不動）。
3. 共用資料夾的**鎖**（止血，平行軌，跟 UI 無關）。

**延後**
- 中央伺服器 / DB、瀏覽器版、整包 app 遷離 Qt（`gui.py` 舊分頁先留著並存）。

**別擋死（保留選項）**
- HTML/JS UI（日後可轉瀏覽器）、引擎乾淨介面（日後可轉伺服器 API）、不把單機假設焊進 UI。

---

## 5. 先做的去風險動作：packaging spike

> 不要用「想的」賭。先花半天做最小驗證，過了再全押。

- 內容：一個 `hello world` 的 pywebview 視窗 + 一個 `js_api` 呼叫 + `pyinstaller` 打包。
- 丟到**真正要部署的那種公司 Windows 機器**跑：WebView2 在不在？exe 包得過嗎？開得起來嗎？
- 過 → 放心走 HTML/pywebview；卡 → 只賠半天，重新評估（Tauri / Electron / 或先把 Qt 精靈做完當過渡）。

---

## 6. 誠實記著的取捨

- **瀏覽器版**：公司部署最爽（零安裝、全員秒更新、IT 友善）——但你不愛那感覺。
- **pywebview 桌面殼**：原生感你要的——但**還是得一台台裝 / 更新**（這點沒比 Qt 好）。
- 走 HTML/JS ＝ 把這取捨**延後、兩扇門都開**；Qt ＝ 現在就替你決定「永遠桌面」。

---

## 7. 低後悔原則

所有 UI 決定都是「**殼**」層；引擎乾淨。押 pywebview 押錯，最壞只是重畫薄殼（換 Tauri / Electron / 退回 Qt），**引擎一行不動**。下檔風險被前面的乾淨架構鎖死在很小。
