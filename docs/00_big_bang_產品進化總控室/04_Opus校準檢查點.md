# Opus 校準檢查點

目的：在進入更高風險的請款批次、狀態機、打包部署前，讓外部 AI 以挑錯角度校準產品方向與資料風險。

## 使用原則

- Opus 只在高風險 checkpoint 進場，不作為每個小功能的日常審查。
- 日常開發先由 Codex 自行完成：實作、測試、文件、人工測試路徑。
- 當下一步會影響資料模型、財務規則、公司部署、輸出架構或使用者主流程時，再整理 checkpoint 請 Opus 校準。
- 已產生的校準請求文件可先當「檢查點便條」，不代表必須立刻呼叫 Opus。

## 建議校準時機

### Checkpoint A：材料價目表 UI + 價格寫入鏈完成後

狀態：適合呼叫 Opus 校準。

原因：

- 已碰到財務資料：單價、金額、請款依據。
- 已有「價目表預設帶價、單據拍照、手動覆蓋不回溯」的核心決策。
- 下一步若做請款批次與狀態機，錯誤設計會更難改。

## 要請 Opus 重點審查

1. 材料價目表 schema 是否足夠支撐專案合約價、歷史價、手動覆蓋。
2. 單價/金額來源欄位是否足以追溯請款依據。
3. 「重跑產出不覆蓋歷史單價」是否有遺漏場景。
4. 請款面板目前自動計算 + 手動覆蓋的規則是否會造成財務風險。
5. 材料價目表 UI 是否對非工程師足夠直覺，是否需要防呆或匯入功能。
6. 下一步做請款批次前，是否還缺必要資料欄位。

## 暫不校準的題目

- PyInstaller 單檔 exe 打包策略。
- 中央伺服器與 SQLite/PostgreSQL 選型。
- 完整角色權限。

這些題目等請款批次與資料模型穩定後再開，不要太早發散。
