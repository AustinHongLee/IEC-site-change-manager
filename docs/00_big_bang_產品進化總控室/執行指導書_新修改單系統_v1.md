## C. 領域知識（你必須懂，否則會做錯）

- **流水號 = 圖號**（1:1 唯一碼）。一張修改單 = 一個圖號；一個圖號可有多張修改單。
- **原始焊口號 = 純數字**（偶有 `w` 前綴）。**不要假設焊口號一定是純數字 `isdigit()`**——修改後會有字母。
- **既有焊口重焊** → `base + 字母(a/b/c…)`；字母 = **第幾次重工**（2 號已焊過，再焊 = `2a`，又一次 = `2b`）。**字母不是操作類型。**
- **全新焊口**（原圖沒有）→ 取高於該流水號所有現有焊口的安全號，慣例 `1000+x`（1001, 1002…），避免撞號。
- **操作只有三種**：裁切 / 加長 / 縮短。加長通常產出「1 口既有重焊 + 1 口全新」；口數**不固定**。
- **焊口管制表**：是「拉樞紐做計量 / NDT」的去正規化底稿，混了 焊口 + 管牙 + 安裝計量，用「屬性.1」欄區分。真實接點只看 `屬性.1 ∈ {焊口, 管牙製作安裝}`。它**不記錄修改原因/類型**——那只活在修改單。對本系統而言它**只用來「查規格 + 驗衝突」，唯讀**。注意：表頭用「**銲**口編號」（金字旁），程式他處可能寫「**焊**口編號」（火字旁），欄名比對要容錯。
- **鐵底必填**：`before + after + 圖號PDF`；其餘（材料、業主簽認等）每案用開關決定必填與否。**注意**：必填與否屬「每案設定 / 完整度層」，**不在 Task 1 的資料層硬編**（見 §G）。
- **scenario（現場慣稱 single / group）**：不是版面差異，是**情境**。資料模型一律用 `scenario = normal / group`（group = 大範圍整改、業主不出圖、承攬商自行修正並記錄）。版面（精簡/多頁）由程式看內容自動決定，**不讓使用者選**。

## D. 設計原則與硬約束

- **canonical 模型在中間，渲染器 / 編號器在邊邊，而且要「笨」**：模型存語意，不存案場字串；字串是輸出。
- **不破壞正在用的工具**：新東西用「新增、並行」的方式做，舊的留著能跑。不做大爆破式取代。
- **後端機件重用、不重寫**：管制表查詢、圖片預處理、標註工具、staging、PDF 算圖等既有功能，後續任務是「包成乾淨介面重用」，不是打掉重做。

### 【Task 1 ✅ 已完成】修改單記錄資料層

### 目標
把 §B 的 schema 變成程式裡的資料類別，並能存/讀 JSON。**純資料層，零 Qt / UI 依賴**，可在無顯示環境被 import 與測試。這是後面所有任務的地基。

### 範圍（只能新增；不得改動現有檔案行為）
- 新增 `control/change_order.py`（若你判斷該放 `control/models/`，在回報裡說明理由）。
- 新增 `tests/test_change_order.py`。
- 若需要小工具，**新增**輔助函式，不修改既有模組。

### 要做的事（具體）
1. 用 `dataclasses` 定義（欄位以 schema v0.2 為準）：
   - `ChangeOrder`（schema_version, id, status, date, series, dwg_no, scenario, welds[], photos[], drawing_pdf, reason, materials[], authorization, audit）
   - `WeldEvent`（joint_type, origin, base, op, rework_index, code, spec{size,sch,material,weld_type}, spec_source）
   - `Photo`（role, file, weld_ref, annotations）、`DrawingPdf`（file, annotations）
   - `Material`（component, component_id, size, schedule, material, qty, unit, remark）
   - `Authorization`（approved_by, approved_at, evidence）
   - `audit` = **稽核軌跡 `history[]`**，每筆 `{who, when, action, detail}`——**不可縮成單一時間戳**（「無 audit_trail」是本專案 P0 地雷）。repo 已有 `operation_journal.py`：先讀它，`history` 設計需與它**相容、不互斥**（Task 1 不用 import，只是別做成日後接不上）。
   - 用 `Enum` 表示固定取值：status(草稿/待補/完整)、scenario(normal/group)、origin(existing/new)、op(裁切/加長/縮短)、role(before/after)、joint_type(焊口/管牙)。
   - **注意**：`joint_type`（焊口 / 管牙＝**接點種類**）≠ `spec.weld_type`（銲接型式，如 BW/SW/RF＝**焊接做法**），兩者勿混。
2. `to_dict()` / `from_dict()`、`save_json(path)` / `load_json(path)`（UTF-8、`ensure_ascii=False`、**原子寫入**：先寫暫存檔再 rename）。
   - **Enum 序列化規則（寫死）**：JSON 一律存 Enum 的 `value`（中文字串，如 `"裁切"`、`"管牙"`），**不存 member name**；`from_dict` 也以 value 反查還原。round-trip 必須穩。
3. **前向相容**：每筆記錄帶 `schema_version`；`from_dict` 要容忍「多出的未知欄位」與「缺少的欄位（給合理預設）」，不可崩。
4. `generate_id(series, date, existing_ids) -> "{series}_{date}_{NN}"`（NN 為當日該流水號的兩位序號，遇重號遞增）。
5. **只做欄位定義 + 存讀 + 結構驗證（型別 / Enum 取值合法）**。**不做業務必填驗證**——「哪些欄位必填、group 要不要圖號PDF」屬於每案設定 / 完整度層（後續任務）；本層所有業務欄位皆**可空（Optional）**，`drawing_pdf` 亦同。**不要**在這裡實作 a/b/1000+ 編號或管制表查詢——那是 Task 2/3。

### 驗收標準
- `pytest` 全綠（含你新增的測試）；**現有測試數不得減少、不得弄壞**。
- 新模組可在無顯示環境 import 成功（無 Qt 依賴）。
