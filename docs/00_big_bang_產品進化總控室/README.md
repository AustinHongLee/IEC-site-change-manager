# Big Bang 產品進化總控室

這裡是「工務修改單」從成熟內部工具進化成公司級產品的核心文件區。

## 核心原則

1. `00_核心文件_產品進化前導書_OPUS.md` 是目前最重要的產品藍圖。
2. 新 AI 進來討論前，必須先讀核心文件與決策紀錄。
3. 發散可以很大，但落地一定要回到 Phase 開發板。
4. 任何大改動都要留下決策原因，避免後續討論重複繞圈。

## 資料夾結構

| 路徑 | 用途 |
|---|---|
| `00_核心文件_產品進化前導書_OPUS.md` | Opus 產出的 Big Bang 級產品前導書，視為主要藍圖 |
| `01_AI討論規則.md` | 多 AI 參與討論時的讀取順序、輸出格式與禁忌 |
| `02_決策紀錄.md` | 已收斂、已採納、已延後或已拒絕的產品/技術決策 |
| `03_Phase0_落地開發板.md` | 第一輪真正要做的工程任務與驗收標準 |
| `04_Opus校準檢查點.md` | 高風險階段完成後，交給 Opus 校準的審查範圍與摘要 |
| `23_DemoSmoke_人工測試入口.md` | 可重跑 demo 專案、範例模板與現場統計輸出的人工測試入口 |
| `ai_sessions/` | 每次 AI 討論、評審、反駁、二次整理的紀錄 |
| `inbox/` | 尚未整理的草稿、截圖、外部建議或臨時筆記 |

## 目前最高優先方向

先做第 0 階段安全收束：

- 啟動守門與資料夾健康檢查
- 單寫者鎖與唯讀模式
- JSON 原子寫入、備份與交易化
- 內建資料一致性稽核
- 修正本機絕對路徑與雲端多機風險

材料明細、請款批次、新 UI、打包優化都很重要，但要排在資料安全地基之後。

目前已建立 demo smoke，可用 `python tools/run_demo_output_smoke.py --output staging/demo_output --overwrite` 產生範例輸出供人工檢查。
