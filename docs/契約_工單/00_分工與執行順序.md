## 0. 分工原則（誰天生該做什麼）

| 角色 | 天生擅長 → 負責 | 天生不該碰 |
|---|---|---|
| **Codex** | 改碼、新增檔案、跑 pytest、**一切 git 操作**、實機打包與 smoke | 需要「看真實資料做判斷」的決策；沒有契約的自由發揮 |
| **Claude（我）** | 任務契約、每片 **pre-push 獨立 review（diff-first）**、驗收判定、領域正確性把關、需要資料計算的產物 | **沙箱內任何 git 寫操作**（FUSE 會弄壞 index，已兩次驗證）；Windows 實機才能驗的事 |
| **使用者** | 視覺驗收（前端/Excel 開起來看）、實機驗證清單（指導書附錄 A）、最終 push、毀滅性操作前的確認 | — |

固定紀律（沿用既有守則，不變）：**一任務一契約 → Codex 先回報 `git diff --stat` 對範圍鎖 → 實作 → 我 review → 過了才 commit（pre-push 閘）**。純加法新檔走大步＋輕量 review；碰既有檔（兩座橋、weld_control、前端大檔、匯出鏈）一律嚴格 review。

## 1. 升級規則（什麼情況從 Codex 換成我動手）

觸發任一條，該片轉我接手（我在雲端改好檔案寫回，Codex 只負責 commit）：

1. 同一片 **兩輪 review 不過**（驗收紅、或改壞範圍外的東西）。
2. `git diff --stat` 超出契約範圍鎖（動到未列檔案，或列內檔案改動量異常放大）。
3. 遇到「要看資料才能裁決」的分歧（料號映射、業主欄位語意、領域規則邊界）——這種不重試，直接丟給我。
4. 打包 spike 類「卡住型」問題：Codex 卡第 2 次就把完整錯誤 log 丟回來，我分析後給修正指令再繼續（不是我接手執行，是我接手診斷）。

## 4. 每片固定流程（貼給 Codex 的作業節奏）

1. 讀 `docs/契約_工單/T{n}_契約_*.md`（沒有契約不開工）。
2. 回報預計改動檔案清單，比對契約範圍鎖。
3. 實作＋跑契約指定測試（最少：指導書 §10.3 第 1 組）。
4. 回報 `git diff --stat` ＋ 關鍵 diff 片段 → 等 Claude review。
5. Review 過 → 一片一 commit（訊息用切片指定的 message）→ 不 push，累積到階段末由使用者 push。
